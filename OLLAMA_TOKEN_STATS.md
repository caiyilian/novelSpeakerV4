# Ollama API 返回 Token 统计字段详解

> 基于 Qwen3:32B 模型（上下文窗口 **40K**）的实际测试结果，服务器 `172.31.102.237:11434`

## 一、如何获取模型的最大上下文窗口

### 方法：调用 `/api/show`

```json
POST /api/show
{"name": "qwen3:32b"}
```

**响应中的关键字段：**

```json
{
  "model_info": {
    "qwen3.context_length": 40960,   // ← 最大上下文窗口（token 数）
    "qwen3.embedding_length": 5120,
    "qwen3.feed_forward_length": 25600,
    "qwen3.block_count": 64,
    "qwen3.attention.head_count": 64,
    "qwen3.attention.head_count_kv": 8,
    "qwen3.attention.key_length": 128,
    "qwen3.attention.value_length": 128
  },
  "details": {
    "parameter_size": "32.8B",
    "quantization_level": "Q4_K_M",
    "family": "qwen3"
  }
}
```

**Qwen3:32B 规格：**
- `qwen3.context_length` = **40960 token = 40K**（不是 32K！）
- 参数大小：32.8B
- 量化方式：Q4_K_M

> ⚠️ 注意：Qwen3:32B 模型卡标注的 32K 是**训练时默认上下文**，但模型架构实际支持 **40960 token（40K）**。Ollama 返回的 `qwen3.context_length` 才是真实可用上限。

---

## 二、每次 API 响应都包含的 Token 字段

Ollama 的 `/api/generate` 和 `/api/chat` 端点，每次请求都会返回以下关键字段：

| 字段 | 类型 | 含义 |
|------|------|------|
| `prompt_eval_count` | int | **输入 token 数**（prompt 文本 + 历史上下文的总和） |
| `eval_count` | int | **输出 token 数**（模型本次生成的回复 token） |
| `context` | int[] | **完整上下文 token ID 数组**，`len(context)` = 当前上下文窗口的实际占用量 |
| `prompt_eval_duration` | int | prompt 处理耗时（纳秒） |
| `eval_duration` | int | 生成耗时（纳秒） |
| `total_duration` | int | 总耗时（纳秒） |
| `load_duration` | int | 模型加载耗时（纳秒） |

### 核心公式

```
本次请求消耗 token = prompt_eval_count + eval_count
上下文窗口使用率 = len(context) / qwen3.context_length × 100%
```

## 三、单轮对话示例（/api/generate）

**请求：**
```json
POST /api/generate
{
  "model": "qwen3:32b",
  "prompt": "你好",
  "stream": false
}
```

**响应（关键字段）：**
```json
{
  "prompt_eval_count": 11,    // 输入 token（prompt "你好" + 特殊 token）
  "eval_count": 188,          // 输出 token（模型回复）
  "context": [151644, 872, 198, ...],  // 完整上下文，长度 = 198
  "total_duration": 17217374823,   // 17.2 秒
  "load_duration": 259023640,      // 0.26 秒
  "prompt_eval_duration": 61198609, // 0.06 秒
  "eval_duration": 4587712669      // 4.59 秒
}
```

**解读：**
- 输入 11 个 token，输出 188 个 token
- 上下文窗口实际占用 **198 个 token**（包含 prompt + 回复 + 特殊 token）
- 上下文使用率：198 / 40960 ≈ **0.5%**

## 四、多轮对话示例（/api/chat）

**请求：**
```json
POST /api/chat
{
  "model": "qwen3:32b",
  "messages": [
    {"role": "user", "content": "你好，请简单介绍一下你自己"},
    {"role": "assistant", "content": "我是 Qwen，由阿里巴巴开发的开源大语言模型。"},
    {"role": "user", "content": "1+1等于几？"}
  ],
  "stream": false
}
```

**响应关键字段：**
```json
{
  "prompt_eval_count": 44,     // 输入 token（含全部历史对话）
  "eval_count": 357,           // 输出 token
  "message": {
    "role": "assistant",
    "content": "1+1等于2。这是最基本的数学运算之一...",
    "thinking": "好的，用户问的是1+1等于几？..."  // Qwen3 特有：思考过程
  }
}
```

**解读：**
- 3 轮对话累积后，单次请求的输入达到 44 个 token
- 输出 357 个 token（含思考过程）

## 五、渐进式多轮 — Context 增长实测

| 轮次 | 消息长度（字符） | prompt_eval_count | eval_count | 上下文增长趋势 |
|------|-----------------|-------------------|------------|---------------|
| 1    | 2               | 11                | 129        | 起步          |
| 2    | 8               | 41                | 292        | +30           |
| 3    | 6               | 94                | 474        | +53           |
| 4    | 8               | 211               | 1028       | +117          |

**规律：** 每轮对话的 `prompt_eval_count` 会随历史消息累积而增长。第 4 轮时单次请求输入已达 211 token。

## 六、流式模式 — done:true 块包含完整统计

流式请求（`stream: true`）时，最后一个 chunk 是 `done: true`，包含完整的 token 统计：

```json
{
  "done": true,
  "done_reason": "stop",
  "prompt_eval_count": 15,
  "eval_count": 234,
  "context": [...],           // 完整上下文 token 列表
  "total_duration": 15437396856,
  "eval_duration": 5713123145
}
```

**流式模式的 `done:true` 块 = 获取实时 token 消耗的最佳位置。**

## 七、Qwen3 特有字段：thinking

Qwen3:32B 在 `/api/generate` 端点会额外返回 `thinking` 字段，包含模型的推理过程：

```json
{
  "response": "你好！有什么我可以帮你的吗？",
  "thinking": "嗯，用户发来你好，这看起来是一个常见的问候语..."
}
```

- `thinking` 的 token **计入 `eval_count`**（属于输出的一部分）
- `/api/chat` 端点的 `message` 对象中也有 `thinking` 字段
- thinking 内容通常比 response 更长（实测 thinking 675 字符 vs response 199 字符）

## 八、上下文窗口边界实测

通过发送不同长度的 prompt，验证模型实际支持的最大上下文：

| prompt 长度（字符） | prompt_eval_count | 是否被截断 |
|---------------------|-------------------|-----------|
| 9,000               | 4,010             | ❌ 未截断  |
| 45,000              | 20,010            | ❌ 未截断  |
| 90,000              | 40,010            | ⚠️ 接近上限 |
| 135,000             | **40,960**        | ✅ 被截断  |
| 180,000             | **40,960**        | ✅ 被截断（不再增长） |
| 225,000             | **40,960**        | ✅ 被截断（不再增长） |

**结论：** `prompt_eval_count` 到达 40960 后不再增长，确认上下文窗口上限为 **40960 token**。

## 九、OpenCode 如何实现实时上下文显示

根据实测数据，OpenCode 实时显示上下文占用百分比的原理：

```
数据来源：
  - prompt_eval_count  → 本次输入 token 数
  - eval_count         → 本次输出 token 数
  - len(context)       → 当前上下文窗口的实际占用量

Qwen3:32B 上下文窗口 = 40960 token（通过 /api/show 获取）

百分比计算：
  context_usage_percent = len(context) / 40960 × 100%
```

**示例：**
- 单轮对话后 `len(context) = 198` → 198 / 40960 ≈ **0.5%**
- 多轮对话后 `len(context) = 211` → 211 / 40960 ≈ **0.5%**
- 当 `len(context)` 接近 40960 时，使用率接近 **100%**，需要截断或重置对话

| 端点 | 方法 | 返回 token 统计 | 备注 |
|------|------|----------------|------|
| `/api/generate` | POST | ✅ prompt_eval_count, eval_count, context | 单轮，支持 Qwen3 thinking |
| `/api/chat` | POST | ✅ prompt_eval_count, eval_count, context | 多轮，message 含 thinking |
| `/api/show` | POST | ✅ model_info.qwen3.context_length | **获取模型最大上下文窗口** |

## 十一、注意事项

1. **`len(context)` 可能为 0** — 某些情况下（如 chat 端点非流式），`context` 数组为空，此时需用 `prompt_eval_count` 近似估算上下文占用
2. **token ≠ 字符** — 中文约 1 字 = 1.5~2 token，英文约 4 字符 = 1 token
3. **`prompt_eval_count` 包含历史** — 多轮对话时，这个值是累积的，不是当前消息单独的长度
4. **特殊 token 也计入** — 如 BOS/EOS 等分隔符 token 会被统计在内
5. **通过 `/api/show` 动态获取** — 不同模型的上下文窗口不同，应通过 `model_info.qwen3.context_length`（或对应模型的 key）动态获取，不要硬编码

# 工具调用（Tool Calling）实战测试

> 测试模型：Qwen3:32B，服务器：`172.31.102.237:11434`
> 测试文本：《狼与香辛料》第一卷（novel.txt，3065行）

## 十二、Ollama 工具调用机制

Ollama 的 `/api/chat` 端点支持 function calling。通过 `tools` 参数定义工具，模型可以：
1. 分析用户请求，决定是否需要调用工具
2. 生成 `tool_calls`（包含函数名和参数）
3. 开发者执行工具，将结果作为 `role: tool` 消息回传
4. 模型根据工具结果生成最终回答

**工具定义格式：**
```json
{
  "type": "function",
  "function": {
    "name": "read_novel_lines",
    "description": "读取 novel.txt 的指定行范围",
    "parameters": {
      "type": "object",
      "properties": {
        "start": {"type": "integer", "description": "起始行号（从1开始）"},
        "count": {"type": "integer", "description": "要读取的行数"}
      },
      "required": ["start", "count"]
    }
  }
}
```

**多步交互流程：**
```
User → 模型(生成 tool_call) → 开发者(执行工具) → 模型(生成 tool_call 或最终回答)
```

## 十四、测试2：第166行「……嗯唔？」— 直接粘贴 vs 工具调用

**任务：** 找出第166行「……嗯唔？」的说话人

**上下文：** 第164行罗伦斯喊"喂！我说你啊！"，第166行"女孩闭着双眼、慢了半拍做出反应"

### 实验 2a：直接粘贴160-170行（11行）

| 指标 | 数值 |
|------|------|
| Token 消耗 | **1,089**（输入 388 + 输出 701） |
| 结果 | ❌ **编造了"卡修（Caro）"** |

**教训：** 上下文不足时，模型会**自信地编造答案**，而不是说"我不确定"。

### 实验 2b：工具调用，不追问（只读155-180行）

| 指标 | 数值 |
|------|------|
| Token 消耗 | **3,594** |
| 结果 | ❌ 只说"女孩"，未给出具体名字 |

**教训：** 模型在范围内找不到名字就放弃了，不会主动扩大搜索范围。

### 实验 2c：工具调用，允许追问扩大范围

| 指标 | 数值 |
|------|------|
| 工具调用次数 | 3 次（155-170 → 171-190 → 191-210） |
| Token 消耗 | **9,453** |
| 结果 | ❌ 扫到210行仍未找到，放弃并说"无法确定" |

**教训：** 模型采用**线性扫描**策略，每次读15-20行，效率低下。名字在第222行，需要更聪明的搜索策略。

### 实验 2d：直接粘贴160-230行（包含第222行"咱的名字是赫萝"）

| 指标 | 数值 |
|------|------|
| Token 消耗 | **2,504** |
| 结果 | ✅ **赫萝** |

### 实验 2e：工具调用 + 跳跃搜索策略（最终成功 ✅）

**优化措施：**
- 在 system prompt 中明确要求"跳跃式搜索"
- 引导模型先读大范围（50-80行）快速定位角色名字
- 提示关键词："名字是"、"名叫"、"我是"、"咱是"

**搜索路径：**
```
Step1: read_novel_lines(163-168)  → 6行，确认说话人是"女孩"
Step2: read_novel_lines(169-218)  → 50行，跳跃搜索 → 未找到名字
Step3: read_novel_lines(219-268)  → 50行，继续跳跃 → 第222行找到"咱的名字是赫萝" ✅
```

| 指标 | 数值 |
|------|------|
| 工具调用次数 | 3 次 |
| Token 消耗 | **10,162** |
| 结果 | ✅ **赫萝** |
| 上下文使用率 | 10,162 / 40,960 = **24.8%** |

**关键发现：** 通过优化 prompt 引导模型采用"跳跃式搜索"策略，成功在 40K 上下文限制内找到了答案。

## 十六、Qwen3:32B 工具调用能力评估

### 优势
- ✅ 支持多步工具调用（实测连续 3 次调用无压力）
- ✅ 能根据工具返回结果推理并给出有据可查的分析
- ✅ 在指令引导下能调整搜索策略（从线性扫描改为跳跃搜索）
- ✅ 上下文窗口 40K 足够支持多轮工具调用 + 复杂推理
- ✅ 诚实——找不到答案时会说"无法确定"，不一定是编造

### 局限
- ⚠️ 默认搜索策略是线性扫描，需要在 prompt 中显式引导跳跃搜索
- ⚠️ 上下文不足时会**自信地编造答案**（实验2a的"卡修"事件）
- ⚠️ 不知道就放弃——如果 prompt 不引导追问，模型会在当前范围内找不到答案就停止

### Prompt 优化建议
1. **明确搜索策略**：告诉模型"不要逐行扫描，要跳跃式搜索"
2. **给出关键词提示**：角色名字通常出现在"名字是"、"我叫"、"咱是"之后
3. **引导主动追问**：告诉模型"如果当前范围找不到，继续扩大搜索"
4. **限制上下文预算**：注意工具调用越多，累积 token 越多

## 十八、多 Agent 协作测试

### 架构设计

针对第166行「……嗯唔？」说话人识别任务，设计了多 Agent 协作架构：

```
                    ┌─────────────┐
                    │    Boss     │
                    │  （主Agent） │
                    │  汇总+决策   │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
        ┌─────┴─────┐ ┌───┴────┐ ┌─────┴─────┐
        │ Worker-1  │ │Worker-2│ │ Worker-3  │
        │ 155-190行 │ │191-230行│ │ 231-270行 │
        └───────────┘ └────────┘ └───────────┘
                           │
                    ┌──────┴──────┐
                    │  Verifier   │
                    │ （验证Agent）│
                    │ 检查幻觉    │
                    └─────────────┘
```

**角色分工：**

| Agent | 职责 | 上下文 |
|-------|------|--------|
| **Worker-1** | 搜索第155-190行，确认说话人身份线索 | 仅该范围文本 |
| **Worker-2** | 搜索第191-230行，寻找角色名字 | 仅该范围文本 |
| **Worker-3** | 搜索第231-270行，确认名字一致性 | 仅该范围文本 |
| **Verifier** | 检查各 Worker 回答是否幻觉 | 各 Worker 的汇总 |
| **Boss** | 汇总所有结果，给出最终回答 | 各 Worker + Verifier 的汇总 |

**关键特性：**
- 所有 Agent 使用同一模型（Qwen3:32B）
- 每个 Agent 有**完全独立的上下文**，互不干扰
- Worker 之间无法通信，各自独立分析
- Boss 不接触原始文本，只接收汇总结果

### 执行流程

**Phase 1：Worker 并行搜索**

| Worker | 范围 | Token 消耗 | 发现 |
|--------|------|-----------|------|
| Worker-1 | 155-190 | 1,841 | 说话人是"女孩"，本范围未找到名字 |
| Worker-2 | 191-230 | 1,886 | 在第222行找到"咱的名字是赫萝" |
| Worker-3 | 231-270 | 1,736 | 确认"赫萝"和"罗伦斯"多次出现 |

**Phase 2：Verifier 验证**

| 指标 | 数值 |
|------|------|
| Token 消耗 | 2,302 |
| Worker-1 评估 | ✅ 可信 |
| Worker-2 评估 | ✅ 可信 |
| Worker-3 评估 | ✅ 可信 |

**Phase 3：Boss 汇总**

| 指标 | 数值 |
|------|------|
| Token 消耗 | 2,930 |
| 最终结论 | ✅ **赫萝** |

### Token 消耗对比

| Agent | Token 消耗 | 占比 |
|-------|-----------|------|
| Worker-1 | 1,841 | 17.2% |
| Worker-2 | 1,886 | 17.6% |
| Worker-3 | 1,736 | 16.2% |
| **Worker 合计** | **5,463** | **51.1%** |
| Verifier | 2,302 | 21.5% |
| **Boss** | **2,930** | **27.4%** |
| **总计** | **10,695** | 100% |

### 与单 Agent 对比

| 方案 | 总消耗 | 主Agent 消耗 | 主Agent 占比 |
|------|--------|-------------|-------------|
| 单 Agent（跳跃搜索） | 10,162 | 10,162 | 100% |
| **多 Agent（5个）** | **10,695** | **2,930** | **27.4%** |

### 关键发现

1. **多 Agent 架构的核心优势：主 Agent 上下文消耗大幅降低**
   - Boss 仅消耗 2,930 tokens，比单 Agent 的 10,162 节省了 **70.5%**
   - 即使 Worker 数量增加，Boss 的上下文消耗也不会显著增长

2. **每个 Agent 有完全独立的上下文**
   - 每个 Worker 调用独立的 API，传入不同的 `messages`
   - Ollama 不会在不同请求之间共享任何状态
   - 每个 Agent 的上下文窗口都是完整的 40K

3. **Verifier 有效防止了幻觉传播**
   - 在之前的单 Agent 测试中，上下文不足会导致模型编造答案
   - Verifier 独立验证各 Worker 的回答，确保结论有据可查

4. **Worker 之间无法协作是双刃剑**
   - ✅ 保证上下文隔离，互不干扰
   - ❌ Worker-2 不知道 Worker-1 已经确认了"女孩"，无法针对性搜索
   - 💡 可以在 Boss 汇总时加入中间发现共享，平衡隔离与协作

5. **总消耗略高于单 Agent**
   - 多 Agent 10,695 vs 单 Agent 10,162（多 533 tokens）
   - 多出的开销来自 Verifier 和 Boss 的汇总请求
   - 但主 Agent 的上下文节省远超这点开销
