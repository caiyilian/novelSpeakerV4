# 上下文窗口利用率提升 & 错误累积风险应对方案

> 日期：2026-06-21
> 
> 基于当前 v2.1 架构，针对两个问题提出改进方案：
> 1. 40K token 上下文窗口利用率不足（当前最多 5K，仅 12%）
> 2. 前几轮错误标注可能通过 ShortMem/CharacterState 传播导致滚雪球

---

## 一、上下文利用率提升方案

### 1.1 现状分析

```
40,960 tokens (100%)  ← Qwen3:32B 完整上下文窗口
│
├── Labeler 输入: ~1.5K tokens (3.7%)
│   ├── 目标对话 + 行号:  ~50
│   ├── ShortMem (5轮摘要):  ~80
│   ├── LongMem (压缩文本):  ~300
│   └── CharacterState:       ~200
│
├── 工具往返 (预期): ~2-3K tokens (7%)
│   ├── read_novel_lines 参数: ~30
│   └── 工具返回原文:          ~500-2000
│
└── 未使用: ~35K tokens (85%+) ← 巨大的浪费
```

结论：**40K 窗口不是"要省着用"的资源，是"不用就浪费"的资源。**

### 1.2 改进方案（按优先级排序）

#### 改进 1：ShortMem 从"存摘要"改为"存原文"（成本最低、收益最高）

**当前：**
```
[短期记忆 - 最近标注]
  #23: 村民 | 询问这是最后一件皮草
  #24: 罗伦斯 | 确认数量并感谢
  #25: 村民 | 感谢罗伦斯
```

**改为：**
```
[最近对话历史]
  #23 [L23] 「这是最后一件了吧？」 → 村民
  #24 [L24] 「嗯，这里确实有……七十件。多谢惠顾。」 → 罗伦斯
  #25 [L25] 「不，我们才要谢谢你呢。只有罗伦斯先生你愿意到这深山里来……」 → 村民
  #26 [L26] 「不过，我也因此拿到上等的皮草啊，我会再来的。」 → 罗伦斯
  #27 [叙事]
  #37 [L37] 「……」 → 罗伦斯
```

**收益分析：**
- 5 轮摘要 80 token → 15 轮原文 900 token。多花的 800 token 对 40K 窗口完全不是负担
- 原文词语本身是标注判断的直接依据——"谁说了什么，用了什么语气，接着是谁回的"
- 连续快速对话中，看到上一句原文比看到摘要准确率高一截

**实现：** ShortMemAgent 改为存 `(line_num, dialogue_text, speaker)` 三元组，`get_summary()` 输出原文而非摘要。

---

#### 改进 2：新增"原文导航"——工具调用前的路标

**问题：** 当前 Labeler 拿到目标行号后，必须调 `read_novel_lines` 才能看到周围内容。而它不知道附近有没有已标注的对话、标注结果是什么——它得读完上下文才能知道。

**方案：** 给 Labeler 一份"导航地图"——标注过的行显示结果，未标注的行标记为叙事/未知：

```
[原文导航 - 目标行附近]
  L155: [叙事] 罗伦斯趴在马车上，几乎快要被睡意吞噬
  L156: [叙事] 
  L160: 「喂！」 → 罗伦斯
  L161: [叙事] 罗伦斯不确定自己有没有喊出声
  L164: 「喂！我说你啊！」 → 罗伦斯
  L165: [叙事] 尽管不忍心，罗伦斯还是打起精神开口说道
→L166: 「……嗯唔？」 ← 【当前待标注】
  L167: [叙事] 随着罗伦斯的声音，女孩闭着双眼
  L170: [叙事]
  L172: 「……」 → 赫萝
```

**收益：**
- Labeler 还没调工具，就看到了上下文的大致结构——叙事/对话分布、最近谁说了什么
- 工具调用从"必须做的盲操作"降级为"确认性阅读"——Labeler 已经大概知道答案，调用工具只为了确认证据
- 减少无效工具往返。v2.1 里 Labeler 可能因为看不到上下行走而反复调用工具读相邻区域

**实现：** Boss 组装输入时，用 `get_dialogue_list()` 和 CharacterState 构造目标行 ±N 行的导航文本。读 `novel.txt` 获取行标记为 `[叙事]`。每条 200 token 可以覆盖 200 行导航（读一页 HTML 的量）。

---

#### 改进 3：CharacterState 加入"最近对话采样"

**当前：**
```
赫萝 (别名: 女孩, 赫萝, 贤狼赫萝)
  关系: 罗伦斯(同行)
  上轮行号: ~第166行
```

**改为：**
```
赫萝 (别名: 女孩, 赫萝, 贤狼赫萝) | 关系: 罗伦斯(同行)
  首次出场: L160 | 最近出场: L172 | 出场次数: 4
  身份状态: 未揭露（罗伦斯仅知其为"女孩"）
  最近对话采样:
    L160: 「喂！」
    L164: 「喂！我说你啊！」
    L172: 「……」
  当前状态: 刚被罗伦斯叫醒，含糊回应中
```

**收益：**
- 3-5 个活跃角色约 1-1.5K token，比当前增加约 800 token
- Labeler 不再只看到一个抽象的角色状态，而是知道角色的**语气、说话风格、当前情绪**

**实现：** `CharacterState` 为每个角色新增 `recent_lines: list[(line_num, text)]` 字段，限制每个角色存最近 3-5 句。

---

#### 改进 4：LongMem 产出双版本（精简版 + 详细版）

**当前：** 一段压缩文本，~300 token。

**改为：**

```
[长期记忆 - 精简版]（~300 token）
  角色：罗伦斯（25岁旅行商人）、赫萝（贤狼，化名女孩）
  主线：离开修道院，在草原遇到赫萝
  别名：女孩 → 赫萝（身份已揭露，见L222）

[长期记忆 - 详细版]（~800 token）
  角色：
    - 罗伦斯：25岁，旅行商人，7年行商经验。性格务实谨慎。
      从村落收了70件皮草后出发，在修道院被骑士盘问后放行。
    - 赫萝（化名"女孩"）：贤狼赫萝，丰收之神。在第160行以"女孩"
      形象首次出场被罗伦斯叫醒。有动物耳朵。在第222行揭露真名为"赫萝"。
    - 骑士：修道院巡逻骑士，第37-56行盘问罗伦斯后放行，后续未再出场。

  关键情节节点：
    - L23-26: 罗伦斯与村民交易70件皮草
    - L37-56: 修道院遇到骑士，被盘问，放行
    - L160-173: 草原马车上叫醒女孩
    - L222: 女孩揭露真名"赫萝"

  别名状态表：
    女孩 = 赫萝（L222已揭露）
    贤狼赫萝 = 赫萝
```

**收益：**
- 详细版多了 500 token，但包含了被精简版压掉的"骑士只是修道院巡逻的"这类可能被误判为新角色的信息
- 当场景切换时（如从修道院到酒馆），精简版可能丢失上一个场景的全部信息，详细版保留了关键脉络

**实现：** LongMem prompt 改为一轮调用产出两个版本，用分隔符标记。

---

#### 改进 5：新增"场景摘要"

**问题：** 标了 700+ 条对话，系统从来不知道当前在哪个场景。修道院还是草原还是酒馆？场景判断对说话人识别非常关键——"修道院里的骑士"和"酒馆里的骑士"可能是不同人。

**方案：**

```
[场景摘要]
  当前场景: L160-173 | 罗伦斯在草原马车上与女孩对话
  上一场景: L37-56  | 罗伦斯在修道院被巡逻骑士盘问
  场景切换:  1次（从修道院到草原）
  当前场景出现角色: 罗伦斯、女孩、骑士（？待确认是否同一人）
```

**收益：**
- 约 100-200 token，极低成本
- 帮助 Labeler 判断"这个骑士是之前那个吗"——场景上下文是线索

**实现：** LongMem 压缩时额外产出场景分段信息。或由 Boss 根据 ShortMem 自动推断。

---

#### 改进 6：工具结果放宽读取量

**问题：** Prompt 建议读 ±20 行（40 行总共）。但工具往返一次就要花费几百 token 的上下文，只拿回 40 行原文有点浪费。

**方案：** Prompt 中建议改为读 ±40-60 行（总共 80-120 行）。一次拿够，减少往返。

---

### 1.3 改进后的预期 Token 消耗

```
改进后的 Labeler 输入（单轮）：
├── 目标对话 + 行号:                   ~50
├── 原文导航 (~100行):                ~1,500
├── 最近对话历史 (15轮原文):           ~900
├── 角色状态 + 最近对话采样:           ~1,200
├── 场景摘要:                          ~150
├── LongMem 精简版:                    ~300
├── LongMem 详细版:                    ~800
└── 工具往返 (1-2次, ~80行/次):        ~3,000
──────────────────────────────────────────
  预计总计:                            ~7,900 (19% 利用率)
```

相比当前的 5K（12%），提升到 ~8K（~19%）。核心变化不是"量"，是**信息密度**——Labeler 拿到的不再是压缩摘要，而是原文、导航、角色采样、场景脉络。

## 二、错误累积风险的应对方案

### 2.1 问题本质

```
第 N 轮 Labeler 标注错误
    ↓
错误结果进入 ShortMem（原文 + 错误的说话人标签）
错误结果更新 CharacterState（错误的别名映射、关系、行号）
    ↓
第 N+1 轮 Labeler 看到：
  "上一轮 #N：「……嗯唔？」→ 罗伦斯"  ← 实际上该是赫萝！
    ↓
受到误导，第 N+1 轮错误概率↑↑
    ↓
滚雪球
```

**核心矛盾：** 不给历史标注 → 上下文太少；给历史标注 → 错误会传染。

### 2.2 解决思路（分层防御）

#### 防御层 1：标注结果标记置信度 / 确定性

不是把历史标注当"铁板钉钉的事实"，而是当"有一定概率正确的参考"：

```
[最近对话历史 - 确定性标记]
  #160 [L160] 「喂！」 → 罗伦斯  [确定性: 高 - 上下文明确有"罗伦斯喊道"]
  #164 [L164] 「喂！我说你啊！」 → 罗伦斯  [确定性: 高 - 同上]
  #166 [L166] 「……嗯唔？」 → 赫萝  [确定性: 中 - 通过排除法，非罗伦斯发声]
```

**方案：** Labeler 在 `<reason>` 中已经有了判断依据。额外输出一个 `confidence` 字段：
```xml
<answer>赫萝</answer>
<confidence>中</confidence>
<reason>上下文显示罗伦斯说了"喂"，然后紧接着"……嗯唔？"，
  从语气和情境推断是女孩回应，但原文未直接指明是女孩发声</reason>
```

Boss 组装 ShortMem 时，低确定性标注加 `[存疑]` 标记。

**关键：** 这样做不是为了"阻止使用"，而是**让后续的 Labeler 在参考时保持警觉**，知道这条标注在判断中权重较低。

---

#### 防御层 2：ShortMem 包含判断线索，不只是结论

**当前 ShortMem 的问题：** 只给了结论（谁说的），没给判断线索。后续 Labeler 只能"信任"或"不信任"这个结论，没有第三个选项。

**方案：** ShortMem 除了标注结果，还保留简短的理由摘要：

```
[最近对话历史]
  #164 [L164] 「喂！我说你啊！」 → 罗伦斯
     理由: L165叙事明确说明"罗伦斯还是打起精神开口说道"
  #166 [L166] 「……嗯唔？」 → 女孩
     理由: 紧接L164罗伦斯对话之后，说话者是L165描述的"女孩"
```

这样后续 Labeler 看到 #164 标了罗伦斯，不仅知道结论，还知道**为什么**是罗伦斯。如果它怀疑这个结论，可以根据理由自行验证。

---

#### 防御层 3：定期交叉验证（懒验证）

不是每轮验证，而是每隔 N 轮（如每 20 轮）让一个 Verifier Agent 回溯最近标注，检查内部一致性：

```
Verifier 的任务：
  "你被要求回溯最近 20 轮标注，检查内部一致性：
   1. 同一角色连续出现时，上下文是否有矛盾？
   2. 别名映射是否有冲突？
   3. 如果前面有低置信度标注，当前上下文是否提供了新的纠正线索？

  输出你发现的所有可疑标注，格式：
  <review>
    #166: 标注为'赫萝'，但彼时上下文尚未揭露真名，
          应标注为'女孩'。建议修正。
    #200: 标注为'骑士'，但场景在酒馆，可能是'佣兵'。
          低置信度，建议重新检查。
  </review>"
```

Verifier 的输出**不直接修改** labeled.txt，而是写入 `review_log.jsonl`。由人类（或更高级别的 Agent）决定是否采纳。

---

#### 防御层 4：CharacterState 的别名映射不靠单次标注更新

**问题：** 如果 Labeler 在第 166 轮错误地把 `<state_update>UPDATE alias: 女孩 -> 赫萝</state_update>` 输出了（而真名实际在第 222 行才揭露），那后续所有轮次都会错误地认为"女孩=赫萝"。

**方案：** 别名映射的更新需要"双重确认"：

```python
def update_alias(self, alias, canonical, evidence_line):
    # 规则1：如果是 identity_reveal（角色自报姓名），需要身份揭露标记
    if "我叫" in evidence or "咱是" in evidence or "名字是" in evidence:
        self.alias_map[alias] = canonical  # 直接更新
    
    # 规则2：如果是推断性映射（Labeler 推理出来的），先记为"待确认"
    else:
        self.pending_aliases[alias] = {
            "proposed": canonical,
            "evidence_line": evidence_line,
            "confirmed": False
        }
        # 在下一次出现该别名且独立确认后才更新
```

这样，L166（"……嗯唔？"）不会触发别名映射更新，因为它不是身份揭露。只有 L222（"咱的名字是赫萝"）才会。这大幅减少了别名映射被错误传播的风险。

---

#### 防御层 5：回溯修正机制

当发现早期标注有错误时，不需要重新标注全部 1349 条。可以让修正 Agent 做"外科手术式"批量修正：

```
修正 Agent 输入：
  "角色'女孩'在第222行之前不应标注为'赫萝'。
   请扫描 labeled.txt 中第1-221行之间的所有标注，
   将所有标注为'赫萝'的条目改为'女孩'。
   只改标注文本，不改 labeled.txt 的其他行。"

修正 Agent 输出：
  需要修改的条目列表 → Boss 执行修改
```

**前提：** 有 Verifier 作为错误检测机制，才有修正的需求。建议先实现防御层 3，再配合修正机制。

---

### 2.3 各防御层的关系

```
防御层 1 (置信度标记)
  └→ 每个标注自带质量评估，历史信息有"使用需谨慎"标记

防御层 2 (ShortMem 含理由)
  └→ 后续 Labeler 不盲目信任历史结论，可以基于理由做独立判断

防御层 4 (别名映射双重确认)
  └→ 阻断最高风险的错误传播渠道（别名映射）
  
防御层 3 (定期交叉验证)
  └→ 事后发现错误（懒验证，不影响实时流程）

防御层 5 (回溯修正)
  └→ 修正已发生的错误（依赖防御层 3 的发现）
```

**实施优先级建议：**

| 顺序 | 防御层 | 改动量 | 效果 |
|:--:|--------|:--:|------|
| 1 | **防御层 4：别名映射双重确认** | 低 | 阻断最高风险传播渠道 |
| 2 | **防御层 2：ShortMem 含理由** | 低 | 提升后续判断质量 |
| 3 | **防御层 1：置信度标记** | 中 | 给系统加上质量感知能力 |
| 4 | **防御层 3：定期交叉验证** | 中高 | 事后纠错机制 |
| 5 | **防御层 5：回溯修正** | 高 | 修复已发生的错误 |

## 三、关于"不给历史 vs 给历史"的权衡

有一个额外的设计角度：**哪些信息可以安全地传给 Labeler，哪些不该传。**

### 安全的（只会帮助，不会误导）：

- **原文本身**（不带标注结果）——"第 164 行说了什么"是中性事实
- **CharacterState 中的出场行号、关系名称**——"罗伦斯出现在第 23、37、164 行"
- **LongMem 中的剧情摘要**——这些是压缩的宏观信息，不太容易被单次错误标注污染
- **场景上下文**——"当前在草原上"

### 危险的（可能传播错误）：

- **ShortMem 中的标注结果**——"#166 → 女孩"如果标错了就是毒药
- **别名映射**——如果提前暴露了赫萝的身份
- **CharacterState 中的状态描述**——如果描述是基于错误标注推断的

### 折中方案：

```
[提示]
  以下为系统记录的过往标注结果，仅供参考。
  如有疑问，请以原文（通过 read_novel_lines 工具读取）为准。
  过往标注中可能存在错误，请独立判断。
```

在 User Message 开头加上这段免责声明。这不是技术手段，但是 Prompt Engineering——告诉 Labeler "你可以质疑历史标注"。配合防御层 2（存理由），Labeler 有能力"看到理由→发现理由不成立→纠正前面的错误"。

## 五、改进前后的输入结构对比
