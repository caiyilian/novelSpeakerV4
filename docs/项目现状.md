## 一、当前架构

### Agent 清单

| # | Agent | 实现方式 | 是否调 API | 工具 | 触发频率 |
|---|-------|---------|:---------:|:----:|:-------:|
| 1 | **Labeler** | `LabelerAgent` 类 | ✅ 每轮 | `read_novel_lines` | 每轮（最多10轮工具循环） |
| 2 | **FactCurator** | `FactCurator` 类 | ✅ 每10轮 | `read_novel_lines` | 每10轮（替代 LongMem 压缩） |
| 3 | **LongMem** | `LongMemAgent` 类 | ❌ 休眠 | 无 | `compress_every=9999` 实质停用 |
| 4 | **ShortMem** | `ShortMemAgent` 类 | ❌ 纯逻辑 | 无 | 每轮更新，保留最近25轮 |
| 5 | **CharacterState** | `CharacterState` 类 | ❌ 纯逻辑 | 无 | 每轮更新 + GPU | 每轮更新 + FactCurator 增量 |

### 数据流

```
每轮标注流程（Boss.process_one）：

 1. 获取 ShortMem（25轮历史对话原文）
 2. 获取 FactCurator 摘要（角色事实库）
 3. 获取 CharacterState（角色状态表）
 4. 构建 Navigation（±20行，无标注嵌入）
 5. 构建 NeighborContext（前后相邻对话，仅作参考）
 6. Labeler 多轮工具循环 → 返回 speaker + reason + summary + state_update
 7. IdentityResolver：已删除（Labeler 自己负责身份搜索）
 8. 更新 ShortMem / CharacterState / save 版本历史
 9. FactCurator 每10轮触发 → 维护结构化角色事实库
10. 写 labeled.txt
```

### 系统参数

| 参数 | 值 | 说明 |
|------|:---:|------|
| ShortMem 轮数 | 25 | 最近25轮对话原文+说话人 |
| LongMem 压缩 | 9999轮 | 实质停用 |
| FactCurator | 每10轮 | 维护角色事实库 |
| Navigation 范围 | ±20行 | 原文导航，无标注嵌入 |
| 工具循环 | 最多10轮 | Labeler 可多次调用 read_novel_lines |
| CharacterState 备份 | 每轮 | 保存到 data/state_history/ |
| 角色名泄露检查 | ✅ | check_roster.py 可用 |

## 三、已完成的修复

### 3.1 Prompt 规则矛盾（修复前50%准确率的主因）

**旧问题：** 规则1（不要编造名字）和规则3（无法判断标非人物发声）被模型合并为"泛称角色→非人物发声"，导致约75个错误。

**修复：**
- 规则1重写：明确"泛称直接作为有效答案"，加入"众人"示例
- 规则2改为"叙事标记优先于推理"，不强调交替规律
- 规则3缩小为"仅在环境声/物体声时使用"
- 规则4/5/6合并为一条

### 3.2 "非人物发声"毒化放大

**旧问题：** `update_character` 在 `parse_state_update` 之前执行，"非人物发声"被注册为真实角色，后续轮次中出现在活跃角色列表。

**修复：** 加 `if speaker != "非人物发声"` 过滤 + NEW character 正则截断描述文本。

### 3.3 LongMem 压缩幻觉

**旧问题：** LongMem 每5轮产出自由文本摘要，产生"赫萝是狐娘""赫萝是精灵"等幻觉信息。

**修复：** `compress_every=9999` 停用，加入 FactCurator Agent（每10轮触发，输出结构化 JSON）。

### 3.4 Navigation 标注嵌入

**旧问题：** 原文导航行中嵌入 `→ speaker` 标注（如 `L160: 「喂！」→ 罗伦斯`），干扰模型判断。

**修复：** 去掉 `→ {sp}`，Navigation 只显示原文内容。

### 3.5 邻居对话干扰

**旧问题：** 要求 Labeler 同时推测前后对话的说话人，导致模型分心、准确率从92%降到76%。

**修复：** 去掉 `<neighbor_check>` 输出要求，neighbor_context 仅作为静默参考。

### 3.6 角色名泄露

**修复：** docstring 中具体角色名替换为抽象描述，新增 `check_roster.py` 检查脚本。

### 3.7 CharacterState 版本备份

**修复：** `save(round_num)` 每次保存前复制旧版本到 `data/state_history/`，保留完整历史。

## 四、剩余问题及下一步

### 当前 Agent 缺口

| Agent | 状态 | 原因 |
|-------|:----:|------|
| Labeler | ✅ | 已稳定 |
| FactCurator | ✅ | 已稳定 |
| **Verifier** | ❌ | 未实现：独立复核 Labeler 的标注，检查罗伦斯↔赫萝交换 |
| **IdentityResolver** | ❌ | 已删除：发现与 Labeler 功能重复，由 Labeler 自己负责 |

### 下一步：Verifier Agent

**目标：** 解决剩余的罗伦斯↔赫萝交换（约占当前错误的 6/17）。

**设计思路：**
- 每轮独立 API 调用，接收 Labeler 的标注 + 原文上下文
- 输出 confirm / flag / reject
- flag 时让 Labeler 重新标注或用 Arbiter 裁决

### 工具调用剩余空间

当前平均 15-16K token/轮（最高 ~20K），40K 窗口剩余充裕。
93% 的轮次有 1 次工具调用，仅 3% 为 0 次——可以鼓励更多轮深度搜索。
