# NovelSpeakerV4 — 项目现状文档

> 整理日期：2026-06-21
> 用途：用于外部咨询，说明当前架构设计、Agent 分工、Prompt 设计、以及 Token 消耗情况

---

## 一、项目目标

对小说《狼与香辛料》第一卷的每一句 `「对话内容」`，自动标注说话人的角色名：

```
输入：  「这是最后一件了吧？」
       「嗯，这里确实有……七十件。多谢惠顾。」
输出：  村民
       罗伦斯
```

### Agent 清单

| # | Agent | 实现方式 | 是否调用 API | 工具 | 触发频率 |
|---|-------|---------|:-----------:|:----:|---------|
| 1 | **Boss** | Python 脚本类 `Boss` | ❌ 纯逻辑 | 无 | 每轮 |
| 2 | **Labeler** | 调用 Qwen3:32B (`/api/chat`) | ✅ 每轮必调 | `read_novel_lines` | 每轮1次（含工具多轮往返） |
| 3 | **ShortMem** | Python 列表维护 `ShortMemAgent` | ❌ 纯逻辑 | 无 | 每轮更新 |
| 4 | **LongMem** | 调用 Qwen3:32B (`/api/chat`) | ✅ 每 N 轮调一次 | 无 | 默认每5轮 |
| 5 | **CharacterState** | Python JSON 文件维护 `CharacterState` | ❌ 纯逻辑 | 无 | 每轮更新 |

## 三、每个 Agent 的详细设计

### 3.1 Boss（调度中心）

**实现：** Python 类 `Boss`，不是大模型调用。

**任务：**
1. 读取 `labeled.txt` 获取已标注数量，计算下一个要标的对话索引
2. 从 `get_dialogue_list()` 获取该对话的行号和内容
3. 从 ShortMem 获取最近 N 轮摘要
4. 从 LongMem 获取压缩后的长期记忆
5. 从 CharacterState 获取当前活跃角色状态
6. **不再读取小说原文**（v2.1 改动）——原文留待 Labeler 自己用工具读
7. 将以上信息组装后发给 Labeler
8. 接收 Labeler 的返回结果，解析 `<answer>`、`<summary>`、`<state_update>`
9. 更新 ShortMem、CharacterState，必要时触发 LongMem 压缩
10. 调用 `write_label()` 写入结果

**输入（给 Labeler 的组装内容）：**
```
第{line_num}行：「{dialogue}」

[短期记忆 - 最近标注]
  #{line}: 说话人 | 摘要

[长期记忆]
  角色：...
  主线：...
  别名映射：...

[角色状态]
  活跃角色名 (别名: ...)
    上轮行号: ~第N行
```

### 3.2 Labeler（标注执行者）

**实现：** 调用 `Qwen3:32B` 的 `/api/chat` 端点，每次调用传入 `tools` 参数。

**工具：** 唯一工具 `read_novel_lines(start, count)`

| 工具名 | 参数 | 返回 |
|--------|------|------|
| `read_novel_lines` | `start`: 起始行号(1-based)；`count`: 读取行数 | `行号: 内容` 格式的文本，最多 `count` 行 |

**System Prompt（完整原文）：**

```
你是一个小说对话角色标注助手。你的任务是判断指定对话的说话人。

你的工具 read_novel_lines(start, count) 可以读取 novel.txt 的任意行范围，
每行以「行号: 内容」格式返回。

**关键规则：你拿不到小说原文。你必须用 read_novel_lines 工具来阅读小说。**

流程：
1. 你只被告知目标对话的行号和内容
2. **第一步必须先调 read_novel_lines** 读目标行附近（±20行）来获取上下文
3. 分析上下文判断说话人是谁
4. 如果第一次读不够（找不到名字线索），再调工具跳到更远范围搜索
   - 角色名字通常出现在自我介绍中（"名字是"、"我叫"、"咱是"、"吾乃"等）
5. 不要在不确定时猜测——用工具去确认

其他规则：
- 角色名字要具体准确——找到该角色在小说中被称呼的具体名字，而非泛指
   （如用具体名字而非"女孩"、"男人"、"商人"等泛称）
- 非对话内容（内心独白、拟声词）标注为"非人物发声"
- 不要编造任何名字！

输出格式（严格遵循，四个标签都要有）：
<answer>说话人名字</answer>
<reason>判断依据，包括关键线索所在行号</reason>
<summary>一句话摘要，格式：说话人 | 做了什么/说了什么</summary>
<state_update>
UPDATE character: 说话人.last_seen_line = {line_num}
（如有别名或关系更新，也在这里声明）
</state_update>
```

**User Message 输入结构（当前版本 v2.1）：**
```
请标注以下对话的说话人：

第{line_num}行：「{dialogue}」

[短期记忆 - 最近标注]
  #{line}: 说话人 | 摘要

[长期记忆]
  角色：...
  主线：...

[角色状态]
  活跃角色名 (别名: ...)
    上轮行号: ~第N行

注意：我没有给你小说原文。请先调用 read_novel_lines 工具读取目标行附近的
上下文，再判断说话人。
```

**输出解析：**
```
<answer>村民</answer>
<reason>第23行是村民询问罗伦斯是否还有货物，对话场景在买卖皮草</reason>
<summary>村民 | 询问这是最后一件皮草</summary>
<state_update>
UPDATE character: 村民.last_seen_line = 23
</state_update>
```

**工具调用流程（循环，最多10轮）：**
```
第1次API调用 → 模型返回 tool_call(read_novel_lines(20, 10))
  → Python 执行工具，返回原文
  → 追加 tool result 到 messages

第2次API调用 → 模型看到原文，可能返回 tool_call 继续搜索
  → Python 执行工具
  → 追加结果

...

第N次API调用 → 模型不再调工具，直接返回最终回答
  → 解析 <answer> 标签
```

### 3.3 ShortMem（短期记忆）

**实现：** Python 列表，不是大模型调用。

**机制：** 维护最近 N 轮（默认 N=5）的标注摘要。每轮标注完成后，将 Labeler 输出的 `<summary>` 追加到列表，超过 N 轮则淘汰最旧的。

**存储示例：**
```
[短期记忆 - 最近标注]
  #23: 村民 | 询问这是最后一件皮草
  #24: 罗伦斯 | 确认数量并感谢
  #25: 村民 | 感谢罗伦斯
  #26: 罗伦斯 | 表示还会再来
  #37: 罗伦斯 | 看到修道院有人挥手
```

### 3.4 LongMem（长期记忆压缩者）

**实现：** 调用 `Qwen3:32B` 的 `/api/chat`，无工具。

**触发：** 每 N 轮（默认 N=5）触发一次压缩。

**System Prompt：**
```
你是一个小说分析记忆压缩助手。将最近的标注摘要压缩为长期记忆。

保留内容：
- 角色关系图谱
- 当前主线情节
- 关键设定
- 角色别名映射

丢弃内容：
- 具体对话内容
- 交易细节
- 路人名字
- 已过时信息

输出格式：
角色：
- 角色名：身份描述

主线：
- 当前进展

别名映射：
- A = B = C
```

**累积策略：** 保留最近 3 次压缩结果拼接，不覆盖（每次追加新压缩结果，保留最近 3 份）。

### 3.5 CharacterState（角色状态表）

**实现：** Python 类 `CharacterState`，数据持久化为 `character_state.json`。

**数据格式（JSON）：**
```json
{
  "characters": {
    "罗伦斯": {
      "aliases": ["罗伦斯"],
      "relations": {"赫萝": "同行"},
      "first_seen_line": 23,
      "last_seen_line": 164,
      "context_summary": "正在招呼马车上的女孩"
    },
    "赫萝": {
      "aliases": ["女孩", "赫萝", "贤狼赫萝"],
      "relations": {"罗伦斯": "同行"},
      "first_seen_line": 160,
      "last_seen_line": 166,
      "context_summary": "刚被罗伦斯叫醒"
    }
  },
  "alias_map": {
    "女孩": "赫萝",
    "赫萝": "赫萝",
    "贤狼赫萝": "赫萝"
  }
}
```

**给 Labeler 的文本摘要（get_state_text）：**
```
[角色状态]
  罗伦斯 (别名: 罗伦斯)
    关系: 赫萝(同行)
    上轮行号: ~第164行
  赫萝 (别名: 女孩, 赫萝, 贤狼赫萝)
    关系: 罗伦斯(同行)
    上轮行号: ~第166行
```

**状态更新来源：** Labeler 输出的 `<state_update>` 标签：
```xml
<state_update>
UPDATE character: 赫萝.last_seen_line = 166
UPDATE alias: 女孩 -> 赫萝
</state_update>
```

## 四、v1（之前版本）的跑测结果

### 4.1 测试规模

- 标注了小说前 **742 条**对话（共 1349 条，完成 55%）
- 备份日志：**554 条**（部分日志因程序启动时清空而丢失）

### 4.2 准确率

| 指标 | 数值 |
|------|:----:|
| 总对话数 | 742 |
| 正确数（修复验证器后） | 481 |
| 错误数 | 261 |
| **准确率** | **64.8%** |

### 4.3 主要错误类型分布

| 错误模式 | 次数 | 占比 | 说明 |
|---------|:----:|:----:|------|
| 罗伦斯 ↔ 赫萝 混淆 | ~135 | 52% | 两人快速对话时角色互换 |
| 新角色出现未识别 | ~40 | 15% | 酒馆场景中采购员/店员等 |
| 称谓粒度差异 | ~30 | 12% | 模型用"米隆商行的卸货工"vs预期"商人\|商贾\|卸货工" |
| 其他 | ~56 | 21% | 零星错误 |

### 4.4 Token 消耗统计（来自 554 条日志）

| 指标 | Labeler | LongMem | 合计 |
|------|:-------:|:-------:|:----:|
| 总消耗 | 1,309,210 | 117,157 | **1,426,367** |
| 出现轮次 | 554/554 | 110/554 | - |
| 平均单次 | **2,363** | **1,065** | **2,575/轮** |

**最大单轮消耗：**

| 排名 | 轮次 | 小说行 | 总计 Token | 占 40K |
|:----:|:----:|:------:|:----------:|:------:|
| 1 | 第145轮 | 第747行 | **4,406** | **10.8%** |
| 2 | 第280轮 | 第1012行 | 4,150 | 10.1% |
| 3 | 第275轮 | 第1004行 | 4,077 | 10.0% |
| 4 | 第325轮 | 第1100行 | 4,063 | 9.9% |
| 5 | 第125轮 | 第707行 | 4,015 | 9.8% |

**单 Agent 最大单次消耗：**
- Labeler：3,444 tokens（第321轮）
- LongMem：1,932 tokens（第145轮）

### 4.5 核心问题：上下文窗口利用率极低

```
40,960 tokens (100%)  ← Qwen3:32B 的完整上下文窗口
    │
    ├── Labeler 输入:    ~2,000 tokens (5%)
    ├── LongMem 输入:    ~1,000 tokens (2.5%)
    ├── 工具结果:        0 tokens（从未调用工具）
    │
    └── 未使用:         ~37,960 tokens (92.5%)
```

**根本原因：** 之前的 v1 版本中，Boss 直接把 ±20 行原文塞进了 Labeler 的输入。模型在输入里已经有了原文内容，**不需要调用 `read_novel_lines` 工具**——工具能读到的信息，输入里已经有了。因此 554 轮中工具调用次数为 **0**。

这也是为什么平均每轮只有 2,575 tokens（不到 40K 的 7%）。

### 4.6 v1 的 Prompt 问题（已修复）

| 问题 | 说明 | 修复方式 |
|------|------|---------|
| **原文写死在输入** | ±20行原文直接给了模型，工具成了摆设 | v2.1 去掉固定上下文，强制用工具读 |
| **工具调用措辞软弱** | "必须尽可能使用"是软约束 | 改为"第一步必须先调"+"每轮至少1次" |
| **Prompt 包含角色名** | 例子里出现"赫萝"是作弊 | 改为通用表述（"具体名字而非泛称"） |
| **`|` 格式乱入** | 多答案用 `|` 分隔是验证格式，不应出现在 prompt | 已删除 |

## 五、当前版本（v2.1）的改动

### 5.1 架构变化

```
v1（已废弃）:                    v2.1（当前）:
  Boss 读原文 → 塞给 Labeler       Boss 不读原文 → Labeler 自己用工具读
  Labeler 从不调工具                Labeler 每轮至少调 1 次工具
  工具调用 = 0                     工具调用 > 0（预期）
  上下文利用率 ~7%                  上下文利用率预期大幅提升
```

### 5.2 结构性改进

1. **CharacterState** — 结构化角色状态表（JSON），替代部分自由文本 LongMem
2. **AliasMap** — 别名系统独立维护，Labeler 直接可查
3. **LongMem 累积式** — 保留最近 3 次压缩历史，不覆盖
4. **验证器修复** — 支持管道分隔标签的匹配

### 5.3 待验证

v2.1 尚未实际跑测。预期改进方向：
- 工具调用次数从 0 上升到每轮至少 1 次
- 上下文利用率上升（更多 token 花在工具往返和结果上）
- 准确率可能先下降（模型需要适应工具调用流程）后提升
