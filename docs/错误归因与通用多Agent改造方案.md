# 错误归因与通用多 Agent 改造方案

更新时间：2026-07-02

## 1. 当前判断

最近一次整卷 API fallback 结果为：

```text
总对话数: 1349
正确:     1266
错误:     83
准确率:   93.8%
```

这说明 API fallback 路径已经基本可用，但系统还没有接近 100%。当前主要瓶颈不再是“模型接口能不能跑”，而是标注流程本身对高风险场景的处理还不够细。

## 2. 哪些不应急着写成代码规则

例如：

```text
梅迪欧商行手下|追兵 -> 梅迪欧商行的人
米隆商行成员 -> 商行的人 / 商行手下
```

这类错误更像是答案粒度或别名问题。它们在当前小说中可以通过扩展标准答案接受项解决，例如把：

```text
【梅迪欧商行手下|追兵】
```

扩展为：

```text
【梅迪欧商行手下|追兵|梅迪欧商行的人】
```

但不应该把“商行”“追兵”等具体背景写成代码规则。这个项目是通用小说对话角色标注系统，换成爱情小说、悬疑小说或校园小说后，这些具体概念未必存在。

因此，具体小说的角色、组织、称谓、别名，应该放在数据或答案层处理，而不是写死进 `src/`。

## 3. 真正需要系统解决的问题

当前更值得优先解决的是通用问题：

1. **明确人物互换**
   - 例如主角二人连续对话时，模型把 A/B 说反。
   - 这不是标签粒度问题，而是说话人判断错误。

2. **具名角色混淆**
   - 例如场景中有多个具名人物，模型误把当前发言归给上一位或更显眼的人物。

3. **非人物/引用/暗号/信件/拟声/心理活动边界**
   - 引号内内容不一定都是人物直接说出口。
   - 有些是信件、暗号、引用词、拟声、未说出口的心理活动。

4. **临时身份描述**
   - 例如“女孩”“男人”“店员”等，需要判断是否已有更具体身份，但不能写死具体小说角色名。

## 4. 原则

后续优化应该遵守：

- 不在代码中硬编码当前小说的角色名、组织名、地名。
- 不写“商行/追兵/某某角色”这类只适用于当前小说的规则。
- 用大模型 agent 做通用推理，而不是用一堆小说特化 if/else。
- 具体答案别名扩展放在数据层或评测层。
- 系统层只处理通用风险：连续对话、引用类型、人物/非人物边界、临时描述、证据不足。

## 5. 通用多 Agent 方案

### 5.1 Labeler

主标注器继续负责首次判断，说出 speaker、reason、summary。

### 5.2 Evidence / Tool Reading

当前工具读取仍由 Python 控制。模型必须通过 `read_novel_lines` 读取小说原文，不能凭记忆或短期上下文直接猜。

### 5.3 Risk Detector

Python 侧做轻量、通用的风险检测，不涉及具体小说名字：

- 当前输出是 `非人物发声`
- 输出是临时身份描述，如女孩、男子、店员等
- 当前处于连续二人短对话，且输出违反短期交替预期
- 目标台词很短、只有语气词、拟声或省略号
- 只做了一次工具读取，且没有明确 speech verb 证据

这些条件只标记“需要复核”，不直接改答案。

### 5.4 Risk Verifier

当 Risk Detector 触发时，调用独立 verifier agent。

Verifier 的职责：

1. 重新读取目标行附近原文。
2. 先判断引号类型：
   - 人物直接发声
   - 引用/信件/暗号
   - 拟声/非人物发声
   - 心理活动或未说出口
3. 再判断说话人。
4. 与 Labeler 的答案比较。
5. 只有在有明确证据时才建议改正。

这样可以重点处理“人物互换”和“非人物/引用边界”，同时避免写小说特化规则。

### 5.5 Final Judge

第一阶段可以暂时不新增复杂 FinalJudge，而是让 Verifier 在高风险场景下给出 confirm/disagree。

后续如果仍有大量争议，可以再加入 FinalJudge：

- Labeler 给出答案
- Risk Verifier 给出复核
- FinalJudge 只在二者冲突时裁决

## 6. 本轮先实施的内容

本轮先做保守实现：

1. 调整 API fallback 优先级：

   ```text
   1. sense-nova/sensenova-6.7-flash-lite
   2. longcat/LongCat-2.0
   3. agnes/agnes-2.0-flash
   4. zhipu/GLM-4-Flash-250414
   5. zhipu/GLM-4-Flash
   6. zhipu/GLM-4.6V-Flash
   ```

2. 去掉不支持工具调用的候选模型。

3. 增加通用 Risk Detector。

4. 对高风险样本调用 Verifier。

5. Verifier prompt 增加通用 quote-type 判断，不写任何具体小说角色名。

## 7. 后续评估方式

下一次整卷或小样本跑完后，应重点看：

- 总准确率是否提升。
- 人物互换类错误是否减少。
- 非人物/引用/暗号边界错误是否减少。
- Verifier 是否引入新的误改。
- `model_calls` 中不同模型调用占比是否符合预期。

如果组织/群体标签粒度仍然占主要错误，应优先考虑扩展答案接受项，而不是写小说特化代码规则。
