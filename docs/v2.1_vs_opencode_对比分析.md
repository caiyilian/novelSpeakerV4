# v2.1 vs opencode-novel-loop 对比分析

> 日期：2026-06-22
> 问题：同样用 Qwen3:32B，opencode-novel-loop 达 ~90% 准确率，v2.1 只有 50%。为什么？

---

## 一、核心差异一览

| 维度 | opencode-novel-loop (~90%) | v2.1 (50%) |
|---|---|---|
| **标注粒度** | 批量标注（一次 5-10 句） | 单句标注（每轮 1 句） |
| **Prompt 长度** | ~200 字，简洁直接 | ~1100 token（5 章节） |
| **原文获取方式** | `read_novel` 读指定范围后直接使用 | `read_novel` 读原文 + 辅助信息全是"可能出错" |
| **前后对话** | 提供前后未标注对话作为线索 | 只提供已标注历史（ShortMem） |
| **角色库** | 轻量显示名/别名表（confidence 标记） | 重 CharacterState（JSON，含死条目） |
| **Verifier** | ✅ 有身份复核（locate/resolve_identity） | ❌ 无 |
| **免责声明** | 1-2 处简洁提醒 | **5 处**"仅供参考、可能出错、不要盲信" |
| **规则设计** | 1 条核心规则 + 举例 | 6 条规则，其中规则 1 和规则 3 互相矛盾 |
| **LongMem 压缩** | ❌ 无（不需要） | ✅ 有（但产生幻觉信息） |

---

## 二、致命差异 #1：批量 vs 单句（最关键）

**opencode 的做法：**
```
一次标注 5 句对话，模型同时看到：
  已标注 #23: 村民「这是最后一件了吧？」speaker=罗伦斯
  待标注 #24: 「嗯，这里确实有……七十件。多谢惠顾。」
  待标注 #25: 「不，我们才要谢谢你呢。」
  待标注 #26: 「不过，我也因此拿到上等的皮草啊。」
  未标注 #37: 「怎么回事？」
```

模型天然能看到对话之间的流转——问→答→感谢→回应，不需要"记忆"。

**v2.1 的做法：**
```
每轮只看 1 句：
  第 24 行：「嗯，这里确实有……七十件。多谢惠顾。」
  
  [短期记忆 - 最近标注]
    #23 [L23] 「这是最后一件了吧？」 → 罗伦斯
  
  [注意：以上标注结果可能包含错误！]
```

每一句都是孤立的"断案"任务。模型必须依赖 ShortMem 中的二手信息，而 Prompt 反复告诉它"这些信息可能是错的"。

**影响：** 批量标注给模型提供了**免费的对话结构信息**。单句标注把"理解对话流"的责任从模型转移到了记忆系统——而记忆系统正是 v2.1 的失败点。

---

## 三、致命差异 #2：Prompt 哲学——"信任" vs "怀疑"

**opencode 的 Prompt 策略：**
```
"相邻对话标签只是连续性线索，如果原文中的'某某说/问/回答'等强证据与标签冲突，以原文为准。"
```

一条规则，清晰明了：**标签是线索，原文是权威。**

**v2.1 的 Prompt 策略：**
```
[辅助信息 1：短期记忆]
  "这些信息来自系统的其他 Agent，不是权威事实。它们可能包含之前轮次标注中的错误。"

[辅助信息 2：长期记忆]
  "压缩过程中可能丢失细节、混淆角色；可能存在过时信息"

[辅助信息 3：角色状态表]
  "别名列表可能不完整或包含错误推断；角色可能存在但未被收录"

User Message 末尾：
  "以下辅助信息由系统其他 Agent 生成，仅供参考，可能有误。"
  "如果与你的原文阅读结果冲突，以原文为准，不要盲信。"
  "我没有把小说原文直接给你。你必须调用 read_novel_lines 工具来阅读小说。"
```

**5 处"不要信"警告**，每条辅助信息都标注了它可能的失败模式。模型被训练成"先怀疑一切，再用工具验证"——但验证完后，它已经消耗了大量 Token，在生成长度受限的情况下选择了最简单的默认答案（罗伦斯）。

**关键洞察：** v2.1 的设计者（我们）过度考虑了"错误传播"，试图通过详细的免责声明让模型保持警惕。但效果适得其反——模型不知道该信什么，最终选择了**统计上最安全的答案**（罗伦斯占 44.7%），而不是**证据上最正确的答案**。

---

## 四、致命差异 #3：原文上下文——"直接给" vs "让模型怀疑着去读"

**opencode：**
```python
# batch_prompt 直接告诉模型读什么范围
f"第一步请调用 read_novel(start_line={context_start}, end_line={context_end}) 读取上下文。"

# 同时提供前后未标注对话作为直接线索
"后续未标注对话（只用于判断当前 batch）："
"- index=37 行号=37 文本=「怎么回事？」"
```

模型知道读哪，而且前后对话直接可见。

**v2.1：**
```python
# 提供原文导航（有标注的行标记 speaker，没标的标叙事）
"  L23: 「这是最后一件了吧？」→ 罗伦斯"
" →L24: 「嗯，这里确实有……七十件。多谢惠顾。」"
"  L25: 「不，我们才要谢谢你呢。」→ (未标注)"
```

Navigation 已经把**标注信息嵌入到原文导航中**了。模型看到 `→ 罗伦斯` 时，Prompt 说的是"可能是错的"。模型陷入认知矛盾：Navigation 这个东西到底能不能信？

**opencode 避免了这个问题：** 它提供原文内容作为"待阅读"的上下文，不把标注结果嵌入其中。标注结果单独放在 `previous_dialogues` 区块，带有明确的"连续性线索"定位。

## 五、v2.1 多出的"有害复杂性"

v2.1 比 opencode 多出的组件，每个都带来了问题：

| 组件 | 设计初衷 | 实际效果 |
|---|---|---|
| **LongMem 压缩** | 保留跨场景的记忆 | 产生幻觉信息（"赫萝是狐娘""赫萝是精灵"） |
| **CharacterState JSON** | 结构化角色追踪 | 被"非人物发声"毒化，死条目污染 |
| **原文导航** | 工具调用前的路标 | 标注结果嵌入原文，造成信任矛盾 |
| **5 处免责声明** | 防止错误传播 | 模型不知道该信什么 → 默认罗伦斯 |
| **System Prompt 5 章节** | 细致的规则指导 | 规则 1 和规则 3 矛盾 → "非人物发声"误用 |

**核心矛盾：** v2.1 试图用**更多组件**来解决"记忆和错误传播"问题，但每个新增组件都引入了自己的失败模式。结果：复杂度增加了 5 倍，准确率下降了 15 个百分点。

## 六、修正后的判断

### opencode 也是 batch-size 1

经确认，opencode-novel-loop 同样使用 `--batch-size 1`，一次只标一句。所以"批量 vs 单句"不是差异原因。

### 上下文充裕但利用方式有问题

40K 窗口峰值只用 24K（60%），确实充裕。open code 的 Prompt 简洁不是因为空间不够，而是设计哲学不同。v2.1 的问题不是"Prompt 太长"，而是 Prompt 的**内容质量**——内部矛盾、过度警告、压缩幻觉。

### 真实的差异来源（重新定位）

| 真正差异 | opencode (~90%) | v2.1 (50%) |
|---|---|---|
| Prompt 内部矛盾 | 无 | 规则 1+3 矛盾导致"非人物发声"误用 |
| 免责声明策略 | 1 处简洁提醒 | 5 处"可能是错的"→ 模型不信任辅助信息 |
| 记忆质量 | ❌ 无 LongMem，没有压缩幻觉 | ✅ 有 LongMem，但产生严重幻觉 |
| ShortMem 形式 | 提供前几句已标注对话（纯原文） | 15 轮原文+理由+speaker，但带"可能出错"标签 |
| Verifier | ✅ 有身份复核 | ❌ 无 |
| 工具调用 | 模型可多次调用 | 99.9% 只调 1 次 |

## 七、修正后的修复方向（保留多 Agent）

### 核心原则
- **保持多 Agent 独立上下文**
- **不追求速度，追求质量**
- **40K 窗口充裕 → 记忆不要压缩太狠**
- **工具调用可以更多轮**

### P0 — 立即修复

1. **修复 Prompt 矛盾**：重写规则 1+3，消除"泛称→非人物发声"的错误推理路径
2. **大幅减少免责声明**：5 处 → 1 处，区分可信/不可信信息
3. **过滤"非人物发声"不入 CharacterState**：修复代码 Bug
4. **修复 NEW character 解析**：截断描述文本

### P1 — 记忆系统改造

5. **LongMem 放松压缩**：
   - 当前：保留最近 3 次压缩（每次 ~200-500 token），约 1000 token
   - 改为：保留最近 5-8 次压缩，或直接保留 ShortMem 的完整原文历史
   - 上下文窗口还有 16K 空余，完全可以放更多信息
6. **ShortMem 扩大**：当前 15 轮 → 可以扩到 25-30 轮
7. **Navigation 去掉标注嵌入**：原文导航中不再显示 `→ 罗伦斯` 等标注结果，只保留行号和内容标记（叙事/对话）

### P2 — 工具调用增强

8. **鼓励更多轮工具调用**：Prompt 中强化"如果第一轮不够，必须追加搜索"
9. **放宽工具读取范围**：±40 行 → ±60-80 行，减少往返需求
10. **加入 Verifier 机制**：参考 opencode 的身份复核流程

---

## 七、结论

**问题不出在模型强度，出在架构哲学。**

opencode-novel-loop 的设计原则：**信任模型 + 信任原文 + 最小化中间层 = 90% 准确率。**
v2.1 的设计问题：**不信任模型 + 不信任辅助信息 + 过多中间层 + 内部矛盾 = 50% 准确率。**

每一次"防错"设计（免责声明、详细规则、LongMem、CharacterState JSON）都在事实上降低了模型做出正确判断的能力。最简单的修复不是加更多规则，而是**删掉大部分"防错"设计**，回到简洁直接的路线。
