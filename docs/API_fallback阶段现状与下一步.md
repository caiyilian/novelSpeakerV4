# API fallback 阶段现状与下一步

更新时间：2026-07-02

## 1. 最初问题

一开始的背景是：本地局域网服务器有 `4090 x2 / 40G`，原本可以跑本项目，但服务器临时要让给同学做实验。于是我们讨论是否可以利用 `E:\projects\free-api\docs\全部免费模型_综合清单.md` 中的免费 API 模型来继续跑小说对话角色标注实验。

当时的约束和目标是：

- 只优先考虑 32K 到 128K 上下文的免费模型；200K+ 上下文模型原则上不动，避免依赖超长上下文能力。
- 需要设置请求频率，避免 API 请求太频繁。
- 需要 fallback：某个模型不可用、限流、超时或工具调用失败时，自动换下一个模型。
- 所有 32K 到 128K 候选模型都不可用时，才真正报错。
- 先用更强或更稳定的外部模型验证系统流程。如果强模型在当前系统下都不能接近 100% 准确率，那么本地小模型 `qwen3:32b` 大概率只会更差。
- 这个项目是通用小说对话角色标注系统，代码中不能硬编码某一本小说的角色名。
- 这一点是硬约束：不能在 `src/` 代码中写入特定小说专有角色名，例如当前测试卷中的罗伦斯、赫萝等。模型应该通过文本证据、记忆和工具读取来识别角色，而不是靠代码里预置答案。

另外，用户明确授权可以使用 opencode 配置中的两个模型：

- `LongCat-2.0`
- `sensenova-6.7-flash-lite`

后来又补充授权了：

- `agnes-2.0-flash`

## 2. 我们实际做了什么

### 2.1 检查模型与 API 可用性

我们先读取了免费模型清单和 opencode 配置，确认了候选模型的来源、baseURL 和 key 位置。测试时发现：

- 普通 chat 能返回，不等于能用于本项目。
- 本项目依赖 tool calling：模型必须能返回标准工具调用，让 Python 工具去读取小说原文。
- 所以健康检查必须覆盖工具调用，而不能只测“回复 OK”。

最初测试中曾误判 LongCat 和 SenseNova 不支持工具调用。后来用更明确的工具调用 prompt 和 `tool_choice` 复测，确认它们都能返回标准 `tool_calls`。

Agnes 一开始直连 401，后来发现是 opencode 配置里的 Agnes key 不是实际可用 key。用户提供的 `D:\Desktop\笔记\cp_paper\word\apikey.txt` 中的 key 可用，随后复制到项目本地私有文件：

```text
config\agnes_api_key
```

该路径已被 `.gitignore` 忽略，不会提交。

### 2.2 新增 API fallback 运行模式

我们在 `src/run_label.py` 中新增了 API fallback 运行模式：

```cmd
python -X utf8 src\run_label.py --provider api-fallback --reset-state --count 1349 --validate
```

默认仍然使用原来的 Ollama 路径。只有显式传入：

```cmd
--provider api-fallback
```

才会启用外部 API 模型池。

### 2.3 控制上下文预算

虽然部分外部模型支持 200K、256K、1M 上下文，但我们决定把 API 模式的上下文预算限制在 40K：

```text
API_CONTEXT_LIMIT = 40000
```

原因是本地目标模型 `qwen3:32b` 约束也是 40K。强模型测试应该验证系统流程，而不是让强模型靠更长上下文作弊。

### 2.4 增加请求间隔和 cooldown

每个模型有自己的 `min_interval`，同一模型连续请求之间会 sleep。遇到 429、5xx、超时等情况时，会进入 cooldown 并尝试下一个模型。

当前策略不是随机，也不是轮询，而是固定优先级 fallback。

### 2.5 修复工具输出过大的问题

第一次 API fallback 试跑时，模型调用了：

```text
search_novel('某个高频角色名')
```

旧版 `search_novel` 会把所有匹配结果全部返回，可能一次返回几十万字符，直接超过 40K 预算。

因此我们给 `search_novel` 增加了：

- 最大命中数
- 最大字符数
- 结果过多时提示模型缩小关键词或改用 `read_novel_lines`

这属于系统级修复，也对将来本地 `qwen3:32b` 有帮助。

### 2.6 兼容不同模型的工具调用形态

有些模型在复杂 prompt 下会把工具调用写成文本形式，例如：

```text
read_novel_lines(36, 5)
```

我们增加了兼容层，将这种文本工具调用转换成标准工具调用结构，再交给现有 Python 工具执行。

### 2.7 增加模型调用记录

一开始的日志没有记录“每条对话实际使用了哪个模型”。这导致无法回头统计不同模型的准确率。

后来我们补充了 `model_calls` 记录。新的日志中每个 agent 会记录：

```json
{
  "provider": "...",
  "model": "...",
  "model_label": "...",
  "tools_enabled": true,
  "tool_choice": "...",
  "tool_calls": 1,
  "coerced_text_tool_call": false
}
```

这样下一轮实验可以按模型统计准确率。

### 2.8 增加小说专有名词泄露门禁

我们把“代码中不能硬编码特定小说角色名”的要求从文档原则升级成启动门禁：

- 私有黑名单放在 `config/forbidden_terms.txt`，每行一个词。
- `config/` 已被 `.gitignore` 忽略，因此这些具体小说名不会进入代码仓库。
- `run_label.py` 启动时会调用 `check_roster.py` 扫描 `src/` 下所有 `.py` 文件。
- 一旦发现代码中出现黑名单词，标注程序直接报错退出，不再继续运行。

当前本机黑名单包含当前测试小说中的角色/组织/专有名词，例如罗伦斯、赫萝、杰廉、马贺特、梅迪欧、米隆、叶勒等。注意这些词只存在于本地私有配置中，不写入 `src/` 代码。

这个门禁的目的不是禁止数据文件里出现角色名；`novel.txt`、`answers.txt`、`labeled.txt` 和日志当然会出现当前小说人物。它只约束通用标注程序本身，防止把某一本小说的答案写死进算法或 prompt。

## 7. 当前错误类型

整卷主要错误模式包括：

```text
贤狼赫萝|赫萝 -> 罗伦斯
罗伦斯 -> 赫萝
梅迪欧商行手下|追兵 -> 梅迪欧商行的人
梅迪欧商行的人|追兵 -> 男子们
列支敦•马贺特|马贺特 -> 罗伦斯
米隆商行同伴|米隆商行成员|米隆商行的人 -> 商行的人
米隆商行的人 -> 商行手下
叶勒 -> 罗伦斯
罗伦斯 -> 叶勒
```

这些错误大致可以分成几类：

1. **主角二人互换**
   - 模型在密集对话、缺少显式说话动词、或者短句接续时，把两人关系判断反。

2. **群体标签边界不一致**
   - 例如“商行成员 / 商行的人 / 商行手下 / 追兵 / 男子们”等。
   - 这可能不是模型能力不足，而是系统没有足够强地约束标签规范。

3. **临时身份与正式姓名切换**
   - 某些角色在首次出现、身份揭示、称谓变化时容易错。

4. **非人物发声 / 信件 / 暗号 / 旁白边界**
   - 这类不是常规人物对话，需要更明确的分类规则。

## 9. 当前阶段结论

现阶段最重要的结论是：

> 外部较强模型在当前系统流程下可以跑到 93.8%，但仍然离“接近 100%”有明显差距。因此主要瓶颈已经不再是 API 接入，而是系统流程、标签规范、记忆使用、验证机制和决策约束。

这也支持最初的判断：

- 先用外部强模型把系统流程压测清楚。
- 如果强模型都不能接近 100%，那直接回到本地 `qwen3:32b` 只会更难。
- 等系统流程更稳后，再优化提示词和流程，让 `qwen3:32b` 在一个更完善的系统下运行。

## 10. 下一步建议

暂时不建议立刻继续大改代码或直接重跑整卷。更合理的下一步是：

1. **停止把“模型能力”当成唯一变量**
   - 这次几乎全由 SenseNova 跑，仍然只有 93.8%。
   - 后续应优先分析错误类型，而不是盲目换模型。

2. **先做错误归因表**
   - 按错误类型分类：主角互换、群体标签边界、身份揭示、非人物发声、称谓合并等。
   - 每类抽样看 10-20 条，确认是 prompt 问题、记忆问题、工具读取不足，还是验证器/标签规范问题。

3. **明确标签规范**
   - 特别是“商行的人 / 商行成员 / 商行手下 / 追兵 / 男子们”这类群体标签。
   - 如果标准答案允许多个别名，系统也应该知道这些别名关系；如果不允许，就要强制模型输出规范名。

4. **增强局部验证**
   - 对高风险场景，例如连续二人对话、称谓切换、多人场景、非人物发声，增加专门校验逻辑。

5. **再回到 qwen3:32b**
   - 只有当外部强模型在同样 40K 预算下接近满分，才说明流程足够成熟。
   - 那时再让 `qwen3:32b` 接入，才能判断小模型能力差距，而不是让它背系统缺陷的锅。


```cmd
set "HTTP_PROXY=" && set "HTTPS_PROXY=" && set "ALL_PROXY=" && set "NO_PROXY=apihub.agnes-ai.com" && python -X utf8 src\run_label.py --provider api-fallback --count 0
```

```cmd
set "HTTP_PROXY=" && set "HTTPS_PROXY=" && set "ALL_PROXY=" && set "NO_PROXY=apihub.agnes-ai.com" && python -X utf8 src\run_label.py --provider api-fallback --api-model sense-nova/sensenova-6.7-flash-lite --reset-state --count 100 --validate
```
