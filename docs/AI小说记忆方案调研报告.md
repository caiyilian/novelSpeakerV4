# AI 小说记忆方案调研报告

> 整理日期：2026-06-21
> 
> 基于对当前项目（NovelSpeakerV4 小说对话角色标注）的记忆设计分析，及两篇知乎文章、外部公开方案的调研。

---

## 一、当前项目（NovelSpeakerV4）的记忆设计

### 架构

```
Python 主控
  ├── Boss         —— 调度中心（只收摘要，不接触原文）
  ├── Labeler      —— 标注执行 + read_novel_lines 工具调用
  ├── ShortMem     —— 最近 N 轮标注摘要（N=5 默认可配置）
  └── LongMem      —— 每 M 轮压缩一次短期记忆（M=5 默认可配置）
```

### 核心机制

- **短期记忆**：维护最近 N 轮标注的摘要。每轮标注完成后，Labeler 输出 `<summary>说话人 | 做了什么/说了什么</summary>`，追加到 ShortMem。
- **长期记忆**：每 M 轮运行一次，将短期记忆中的摘要发给 Qwen3:32B，用 system prompt 指导其压缩提炼为：角色关系、主线情节、关键设定、角色别名映射。输出结果覆盖 `LongMem.long_memory`。
- **Boss 的输入**：每轮标注前，Boss 取 ShortMem 摘要 + LongMem 摘要 + 上下文片段，组装后发给 Labeler。

### 当前方案的优点

1. **简单明确**：两级记忆，边界清晰，容易理解和调试
2. **Pyth精确数字计算on 控制流**：Agent 不写文件，所有状态由 Python 管理
3. **渐进压缩**：LongMem 的压缩频率可配置，近期信息保留细节，远期信息被压缩
4. **质量不错**：前 10 条 90% 正确率，从第 5 条起 100%

### 当前方案的局限 / 改进空间

| 问题 | 说明 |
|------|------|
| 记忆维度单一 | 只区分为"短期和"长期"，没有按**内容类型**分层（角色信息 vs 情节信息 vs 别名映射 混在同一个 LongMem 字符串里） |
| 压缩是**替换式**而非**累积式** | LongMem 每次压缩后直接覆盖上一次的结果，可能出现"漂移"——关键信息在一次次压缩中丢失或变形 |
| 没有**角色状态追踪** | ShortMem 只记录了"谁说了什么"，但没有显式维护"赫萝当前是什么身份状态""罗伦斯和赫萝的关系进展到哪一步" |
| 没有**伏笔/线索追踪** | 标注任务不需要伏笔，但如果扩展到更复杂的叙事任务，就需要 |
| 记忆更新被**嵌入标注流程** | ShortMem 和 LongMem 都是 Labeler 输出的副产品，没有**独立的记忆维护 Agent** |
| 缺乏**分层归档** | 只分两档（活跃 / 压缩后），没有"不活跃但保留"的中间层 |

## 二、参考方案一：12 Agent 小说流水线的记忆体系统

> 来源：知乎「javascript」《AI写长篇小说最大的坑不是文笔，是"失忆"——我是怎么用记忆体系统解决的》
> 
> 作者构建了 12 个 Agent 的小说流水线，其中记忆体是独立维护的子系统。

### 核心架构：七层文件系统

作者将记忆拆分为 **7 个独立的追踪文件**，每个文件由专门的 Memory Agent 维护：

| 层级 | 文件/名称 | 内容 | 更新频率 |
|------|-----------|------|----------|
| 1 | **角色状态表** | 境界等级、身份、当前关系状态 | 每章更新 |
| 2 | **章节摘要（MEMORY.md）** | 最近 5 章详细摘要（活跃记忆） | 每章更新 |
| 3 | **系统/金手指状态表** | 功能解锁进度、消耗、余额 | 每章更新 |
| 4 | **伏笔追踪表** | 编号、内容、铺设章节、状态（待处理/已推进/已回收）、预计回收卷 | 每章扫一遍 |
| 5 | **资源清单** | 灵石、丹药、法宝、功法等物品收支 | 每章更新 |
| 6 | **势力关系图** | 树状图维护角色关系网络 | 角色关系变化时更新 |
| 7 | **时间线** | 故事内时间推进记录 | 每章更新 |

### 关键设计：分层归档

```
活跃记忆（最近5章完整摘要）
    ↓
归档记忆（每10章一组，压缩成 ≤100 字的概括）
    ↓
审计 / 写作时合并使用
```

Writer 写第 50 章时的输入组合：
- 第 50 章大纲
- 第 45-49 章详细摘要（活跃记忆）
- 第 1-44 章压缩摘要（归档记忆）
- 当前角色 / 系统 / 伏笔状态表

### Memory Agent 的"禁止事项"

1. **禁止添加主观评价**——记忆体只记事实，评价是 Reader/Auditor 的事
2. **禁止包含未来剧情信息**——只记"已经发生的事"，不记大纲计划，防止剧透
3. 其他 5 条禁令围绕客观性和准确性

### 实际效果（20 章验证）

| 拦截的问题 | 发现章节 | 触发机制 |
|-----------|----------|----------|
| 主角境界描述不一致 | 第 14 章 | 角色状态表发现灵力总量写错 |
| 伏笔断裂 | 第 19 章 | 伏笔追踪表提醒 F003 已 7 章未推进 |
| 天命点算错 | 第 5 章 | 系统状态表发现推演消耗和余额对不上 |
| 角色关系写反 | 第 16 章 | 势力关系图发现中立角色被写成敌对 |

### 对你的项目的借鉴意义

- **按内容类型分层**：不要把角色信息、情节信息、别名映射全混在一起，用独立的字段/文件分别追踪
- **独立记忆 Agent**：Memory 不是 Labeler 的副产品，而是一个主动的、有自己规则和工具的 Agent
- **状态表 > 摘要**：结构化追踪（状态字段）比自由文本摘要更可靠，LLM 不容易"篡改"已有数据
- **伏笔追踪原型**：即使是对话标注任务，如果加入"每条对话的上下文证据链追踪"，就等于一种微型伏笔系统
- **禁止主观评价**：ShortMem/LongMem 的压缩过程中如果 LLM 加入了主观判断，会污染后续 Labeler 的推理

## 三、参考方案二：文件系统 + 工序固化 + 多 Agent

> 来源：知乎「大模型写长篇总"失忆"？一文解决AI小说的记忆难题」
> 
> 从"金鱼记忆"到"可控交付"——让 AI 写出 200 章不崩坏的长篇小说。

### 方案一：文件系统记忆（Cursor + Me2AI）

用 **多层 Markdown 文件** 将记忆物理隔离，写每章时强制加载：

```
项目根目录/
├── chapter/              # 正文（每章 .md）
├── SPEC/Me2AI/          # 人对 AI 的硬约束（写作规则）
│   ├── 写作风格.md        # 禁词、句式、章节结构
│   ├── 故事整体设定.md    # 时代边界、世界规则
│   ├── 核心角色.md        # 能力/限制/成长方向
│   └── 故事主线设计和进展.md
└── SPEC/AI2AI/          # AI 自动管理的项目台账
    ├── 章节目录以及写作状态.md
    ├── 历史人物与时间线校验.md
    ├── 中长线伏笔设计和进展.md
    └── 各角色线设计和进展.md
```

**写每章时的上下文加载规则**：
- Me2AI 下所有设定文件（不变）
- AI2AI 下所有台账文件（每章更新）
- 上一章 + 上两章正文

**核心理念**：Me2AI = 人类写的不变设定，AI2AI = AI 自动维护的动态台账。两者界限分明。

### 方案二：codebubby 固化四道工序

把"写一章"固化成四道不可跳过的工序：

| 工序 | 内容 | 关键要求 |
|------|------|----------|
| A：章内计划 | 核心问题 + 3-5 个可验证证据载体 + 必付代价 | 写出具体物件、代价 |
| B：生成正文 | 动词优先，控制形容词 | 删空话连接词 |
| C：反向摘要 | AI 读自己写的正文，提取摘要 | 排除"我以为写了但实际上没写" |
| D：签约级润色 | 删心理总结、空话、回环对话 | 改为"动作+证据+代价" |

每章写完立刻更新台账（AI2AI 文件）。

### 方案三：StoryWriter 多智能体框架（学术方案）

学术界提出的多 Agent 框架，三个专用 Agent：

```
OutlineAgent  →  生成基于事件的大纲（事件情节 + 人物关系）
     ↓
PlanningAgent →  细化事件，决定哪些事件写入每章
     ↓
WritingAgent  →  根据当前事件动态压缩历史，写正文
```

思路借鉴：**把写作任务拆解给不同智能体，各自负责一部分记忆**。

### 实践建议：三层记忆架构

```
短期记忆  →  当前章节上下文（Cursor 管理）
中期记忆  →  人物设定、世界规则（Me2AI 文件固化）
长期记忆  →  伏笔进度、角色线进展（AI2AI 台账追踪）
记忆控制器 →  codebubby 工序（确保每步不遗漏）
```

已在番茄小说《一纸洛阳》验证有效，50 章后仍保持一致性。

### 拓展方向

- **RAG 增强写作**：向量数据库存储所有历史章节，每次写作前检索最相关的 5-10 个片段
- **KV 缓存优化**：利用 LLM 的 KV 缓存技术重用历史中间状态
- **压缩式记忆**：用小模型定期对历史章节摘要

### 对你的项目的借鉴意义

- **Me2AI/AI2AI 的二分法**：项目中"当前项目配置/规则"是 Me2AI 类的（不可变），"角色状态"是 AI2AI 类的（动态更新）。可以显式区分
- **工序固化（C 工序：反向摘要）**：Labeler 标注后，让另一个 Agent 读 Labeler 的输出和原文，做一次"反向验证"——这个标注真的合理吗？
- **StoryWriter 的 Outline→Planner→Writer 流水线**：对应到标注任务 = 角色图谱 Agent（维护角色关系）→ 上下文检索 Agent（找相关原文）→ 标注 Agent（做判断）

## 四、外部方案：MemGPT / Letta —— 操作系统式记忆管理

> MemGPT (Memory-GPT) 是 UC Berkeley 提出的开源项目，后来更名为 Letta。
> 
> 核心理念：**把 LLM 的上下文窗口当做操作系统内存来管理**。

### 架构

```
┌──────────────────────────────────────┐
│      LLM 上下文窗口（"主存"）      │
│  ┌────────────┐  ┌────────────────┐ │
│  │ 系统提示   │  │  工作记忆      │ │
│  │（持久指令）│  │ （当前对话内容）│ │
│  └────────────┘  └────────────────┘ │
└──────────────┬───────────────────────┘
               │ 函数调用：内存读/写
    ┌──────────┴──────────┐
    │   外部存储（"外存"） │
    │  ┌────────────────┐ │
    │  │  归档记忆       │ │
    │  │ （向量检索 +    │ │
    │  │   分页存储）    │ │
    │  └────────────────┘ │
    │  ┌────────────────┐ │
    │  │  会话历史       │ │
    │  └────────────────┘ │
    └─────────────────────┘
```

### 关键技术

1. **虚拟上下文**：LLM 以为自己有无限上下文，实际上下文窗口只装当前需要的"页面"
2. **自主内存管理**：LLM 自己决定什么时候把信息写入外部存储、什么时候检索
3. **函数调用驱动**：通过 `core_memory_replace`、`archival_memory_insert`、`archival_memory_search` 等函数调用操作内存
4. **分页 + 语义检索**：长的记忆被分页存储，检索用向量相似度

### 对你的项目的借鉴意义

- **短期 vs 长期 vs 归档**的三级模型比你现在两级更细粒度
- **LLM 主动管理记忆**：不是被动压缩，而是 LLM 自己判断什么该记、什么该检索
- **语义检索替代文本拼接**：LongMem 可以用向量检索的方式，每次只把最相关的记忆片段喂给 Labeler，而不是整段压缩文本
- **核心记忆（不可变）vs 归档记忆（增长型）**：角色基本设定 = 核心记忆，标注历史 = 归档记忆

## 五、外部方案：结构化记忆（知识图谱 / 树状记忆）

### 什么是"树状记忆"

本质是 **用结构化数据表示叙事元素及其关系**，而不是用自由文本。

```
树状结构示例：
  角色
    ├── 罗伦斯（25岁旅行商人）
    │   ├── 当前状态：刚从修道院离开
    │   ├── 关系
    │   │   ├── 赫萝（同行伙伴，第N章相遇）
    │   │   │   └── 别名：女孩、贤狼赫萝、丰收之神
    │   │   └── 骑士（第X章遇到，敌对关系）
    │   └── 随身物品：马车、皮草
    └── 赫萝
        ├── 当前状态：身份未揭露
        └── 别名映射：[赫萝, 女孩, 贤狼赫萝, 丰收之神]
```

### 为什么比自由文本好

| 优点 | 说明 |
|------|------|
| LLM 不能"改写"已有事实 | 自由文本压缩时可能漂移，结构化字段是显式更新的 |
| 可精确查询 | "赫萝有哪些别名"可以用字段精确返回，不需要 LLM 从一段文本中再解析 |
| 易于合并 / 去重 | 多条标注摘要中提到同一个角色，可以合并更新同一个字段 |
| 支持关系推理 | "骑士→敌对→罗伦斯→同伴→赫萝" 可以在关系图上直接推导 |

### 几种结构化记忆形态

| 形态 | 适用场景 | 举例 |
|------|----------|------|
| **JSON 对象** | 固定字段的状态追踪 | `{"赫萝": {"aliases": ["女孩","贤狼赫萝"], "current_role": "同行者"}}` |
| **知识图谱** | 实体关系复杂、需要图遍历 | `(罗伦斯) -[同行]-> (赫萝) -[真实身份是]-> (贤狼)` |
| **树状文件结构** | 人类可读、版本控制友好 | 上面例子中的 YAML/Markdown 目录树 |
| **关系数据库 / SQLite** | 大量结构化检索 | `SELECT * FROM character_aliases WHERE character='赫萝'` |

### 对你的项目的借鉴意义

- 标注任务中最核心的记忆类型其实是 **角色别名映射** 和 **角色关系状态**，这两类用结构化存储比自由文本效果好得多
- 考虑维护一个 `character_state.json`：
  ```json
  {
    "characters": {
      "罗伦斯": {
        "aliases": ["罗伦斯"],
        "relations": {"赫萝": "同行"},
        "last_seen_line": 164,
        "current_status": "正在招呼女孩"
      },
      "赫萝": {
        "aliases": ["女孩", "赫萝", "贤狼赫萝"],
        "identity_revealed": false,
        "identity_revealed_line": null,
        "relations": {"罗伦斯": "同行"},
        "last_seen_line": 166
      }
    }
  }
  ```
- 这个 JSON 可以替代当前的 LongMem 字符串，Labeler 可以直接读取结构化字段而非解析一段自由文本

## 六、外部方案：RAG（检索增强生成） — 用向量检索替代记忆压缩

### 核心思路

不做记忆压缩，而是把所有历史标注摘要存入向量数据库，每次标注时检索最相关的 K 条。

```
每次标注
    ↓
将当前对话转为向量 → 在历史摘要中检索最相似的 K 条
    ↓
把检索到的 K 条摘要 + 当前对话上下文 → 发给 Labeler
```

### 优点

- **不失真**：不需要压缩，历史信息原文保留
- **按需检索**：只喂最相关的上下文，不是"全部"或"压缩后"
- **天然支持长距离指代**：如果第 200 条对话提到"那个农夫"，可以检索到第 50 条对话中"农夫"首次出场

### 缺点 / 挑战

- 需要额外的基建（向量数据库 / embedding 模型）
- 检索准确率依赖 embedding 质量
- 对"需要全局视野"的判断（如角色别名映射）效果不如结构化的显式状态表

### 对你的项目的适用性分析

| 适合 RAG | 不适合 RAG |
|----------|------------|
| 长距离指代：找"这个农夫"之前在哪出现过 | 角色别名映射：赫萝=女孩=贤狼赫萝 |
| 相似场景推理：类似的对话模式之前是谁说的 | 当前剧情阶段：罗伦斯在修道院还是草原上 |
| 上下文回溯：某行附近的原文 | 角色关系维护：谁和谁是敌对/朋友 |

**建议**：RAG + 结构化状态表的组合。RAG 处理自由文本检索，结构化状态表维护实时变动的元数据。

## 七、方案对比总结

| 方案 | 核心机制 | Token 效率 | 信息保真度 | 实现复杂度 | 适用场景 |
|------|----------|-----------|-----------|-----------|----------|
| **当前方案**（ShortMem + LongMem） | 两级压缩 | 中 | 中（压缩漂移风险） | 低 | 中小规模标注 |
| **知乎文章1**（七层文件 + 归档） | 多维度独立追踪 + 分层归档 | 中高 | 高（结构化文件） | 中高 | 长篇写作，需要追踪多类信息 |
| **知乎文章2**（Me2AI/AI2AI + 工序固化） | 文件系统分类 + 强制性上下文加载 | 中 | 高（文件是源） | 中 | 多人协作写作，需要版本控制 |
| **MemGPT/Letta** | 操作系统式上下文管理 + 自主内存读写 | 高 | 中高 | 高 | 需要 LLM 自主管理记忆 |
| **结构化记忆**（JSON/知识图谱） | 显式字段更新 | 最高 | 最高（无漂移） | 低-中 | 角色状态追踪、别名映射 |
| **RAG**（向量检索） | 语义检索替代压缩 | 中高 | 最高（原文保留） | 中 | 长距离指代、上下文回溯 |

## 八、对你的项目（NovelSpeakerV4）的具体改进建议

### 建议 1：增加结构化状态表（最优先、成本最低）

在当前基础上，新增一个 `character_state.json` 或 Python 字典：

```python
character_state = {
    "characters": {
        "罗伦斯": {
            "aliases": ["罗伦斯"],
            "relations": {},
            "last_seen_line": 0,
            "current_context": ""
        }
    },
    "pending_identity_reveals": {
        # "行号: 当前称呼 → 真实身份"
        "222": {"alias": "女孩", "real_name": "赫萝"}
    }
}
```

Labeler 的输出除了 `<answer>` 和 `<summary>` 外，增加一个 `<state_update>` 字段：
```xml
<state_update>
  UPDATE character: 罗伦斯.last_seen_line = 166
  UPDATE character: 女孩.add_alias("赫萝")
  SET reveal: 166行 女孩→赫萝
</state_update>
```

### 建议 2：LongMem 改为累积式 + 逐类压缩

当前的 LongMem 是**替换式**（每次压缩后覆盖上一次结果），建议改为**累积式**：

- 角色信息单独存储，不参与普通压缩（角色状态是不可丢失的核心信息）
- 情节摘要用渐进压缩，但保留一个 `compression_history` 列表方便回溯
- 定期让一个验证 Agent 检查 LongMem 是否与 ShortMem 的历史摘要矛盾

### 建议 3：引入"反向验证"工序（借鉴知乎文章2的 C 工序）

增加一个 **Verifier Agent**，在 Labeler 标注后，读标注结果 + 原文上下文，做一次"合理性检查"。这不同于当前的 `--validate`（那是和 ground truth 对比）。Verifier 是盲测——只看原文和标注，判断标注是否合理。

### 建议 4：角色别名系统独立维护

将角色别名映射从 LongMem 文本中剥离出来，单独存储：

```python
alias_map = {
    "赫萝": ["女孩", "赫萝", "贤狼赫萝", "丰收之神"],
    "骑士": ["骑士", "修道院骑士"],
    "村民": ["村民", "农夫"]  # 同一人物跨场景的别名
}
```

Labeler 遇到可疑身份时优先查别名表，而不是重新推理。

### 建议 5：工具增强（可选的 RAG 试水）

给 Labeler 新增一个 `search_similar_scenes(dialogue)` 工具——内部实现可以先用简单的关键词匹配，效果不好再上向量检索。让 Labeler 在不确定时主动搜索"类似对话此前是谁说的"。
