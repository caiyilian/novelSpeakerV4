## 三、根因分析

### 根因一：Prompt 规则矛盾 → "非人物发声"灾难性误用（~75 个错误）

**代码位置：** `src/run_label.py:596-612`（规则 1 + 规则 3）

**问题链路：**

```
System Prompt 中的两条规则：

规则1 (§4.1): "如果原文中一个角色只有泛称，不要自己起名字"
    ↓ （模型的理解：泛称 = 不能用）
规则3 (§4.3): "如果你读了很多上下文仍无法确定，标注为'非人物发声'"
    ↓
模型错误合并：泛称角色（采购员、追兵、店老板…）
    = "没有具体名字的角色"
    = "无法确定的角色"  
    = 标"非人物发声" ← 错误！
```

**实际案例：**
- L24「嗯，这里确实有……七十件。多谢惠顾。」→ 标"非人物发声"（应为"村民"）
- 模型在 reason 中自己写道："原文中村民始终未被赋予具体名字……根据规则6，应标注为'非人物发声'"
- L1123「啊，哇，没问题！就请以两百一十枚卖给本行！」→ 标"非人物发声"（应为鉴定员/采购人员）

**放大机制（代码 Bug）：**

`src/run_label.py:888` 行在解析 `<state_update>` **之前**就执行了：
```python
self.char_state.update_character(speaker, ...)  # speaker="非人物发声"时也被注册
```

导致"非人物发声"被注册为活跃角色（见 `character_state.json:53-83`），后续轮次中 Labeler 看到它在角色列表里 → 更倾向输出它 → **自我强化循环**。

### 根因二：罗伦斯↔赫萝快速对话交替归属失败（114 个错误）

**现象：** 104 次赫萝被标为罗伦斯，10 次反向。

**成因分析：**

这不是 Prompt 设计缺陷，而是模型推理能力的瓶颈。两人密集对话时，模型过度依赖"对话交替规律"（A→B→A→B）来推断说话人，忽略了原文中的叙事标记。

**关键约束（用户指出）：** 很多情况下**同一角色会连续说多句**，不满足交替规律。如果 Prompt 中过于强调交替规律，模型可能错误地认为"上一轮说话的人这一轮不可能再说话"。

**正确的判断优先级应该是：**

1. **叙事标记优先：** "XX说"、"XX开口道"、"XX喊道"——这是权威信号，必须优先于交替规律
2. **交替规律作为辅助：** 仅在叙事中**没有**明确指明说话人时，交替规律才作为推测手段
3. **同一人连续说话：** 叙事中通常会有标记（"他又补充道"、"XX继续说"），否则可能是不分段的连续台词

**对比 v1：** v1 时期此错误占 52%，v2.1 降到 38.1%。工具调用（模型能读原文）有帮助但未根除。

### 根因三：CharacterState 角色名污染（22 个死条目）

**现象：** `character_state.json` 中出现大量无效角色条目：

| 污染条目 | 来源 | 问题 |
|---|---|---|
| `寄宿在麦子里的狼（原文揭露）` | Labeler 的 NEW character 声明 | 描述被当成角色名 |
| `农夫 —— 帕斯罗村麦田劳作的村民，性格淳朴` | 同上 | 长描述被当成独立角色 |
| `杰廉 —— 初入行的旅行商人，缺了一颗门牙` | 同上 | 与"杰廉"重复 |
| `叶勒 —— 帕斯罗村农民，今年负责收割的主角...` | 同上 | 与"叶勒"重复 |
| `无` | 空值被注册 | 完全无效 |

这些死条目 `first_seen_line: null`、`last_seen_line: null`，永远无法匹配。但它们挤占角色列表空间、增加 token 消耗。

**代码位置：** `src/run_label.py` 中 `parse_state_update()` 方法，`NEW character` 的解析正则未截断描述文本。

---

### 根因四：泛称/别名替换不一致（22+ 个错误）

| 现象 | 次数 | 原因 |
|---|---|---|
| "男子" → 全标为"丈夫" | 9/9 (100%) | 模型倾向使用已在 CharacterState 中出现的称呼 |
| "列支敦•马贺特" → 全标为"马贺特" | 5/5 (100%) | CharacterState 只登记了简名"马贺特" |
| "农夫" → 全标错 | 2/2 (100%) | 模型不确认"农夫"是否等于某个已知角色 |

**根因：** Labeler 收到的 CharacterState 中只有简称/常见名，当原文使用全名或不同称呼时，模型倾向替换为已知条目中的名字。Prompt 规则 1（"用当前语境中的称呼"）未被严格遵守。

---

### 根因五：模型对"罗伦斯"有系统性偏好（新发现）

**现象：** 除赫萝外，几乎所有其他角色都被大规模误标为罗伦斯：

| 被标为罗伦斯的角色 | 次数 |
|---|---|
| 赫萝 | 192 |
| 马贺特 | 47 |
| 非人物发声 | 33 |
| 杰廉 | 13 |
| 寄宿在麦子里的狼 | 10 |
| 怀兹 | 10 |

这说明模型在不确定说话人时，存在一个**"默认输出罗伦斯"的 hidden bias**。罗伦斯是小说第一主角，模型可能隐式推断"主要角色在场 → 对话更可能是主角说的"。

**根因：** 
1. 罗伦斯是 CharacterState 中第一个注册的角色（`first_seen_line: 23`），在角色列表中排在首位
2. 罗伦斯出现频率最高（423 次正确输出），模型形成了"罗伦斯是安全默认值"的统计偏好
3. ShortMem/Navigation 中积累的早期错误（罗伦斯↔赫萝交换）通过辅助信息持续强化这个偏见

---

### 根因六：工具调用"一击即止"

1349 轮中，93% 有工具调用（vs v1 的 0%），但只有 **1 轮**调用了 2 次工具。模型倾向于一次读 ±40-80 行就下结论，即使信息不足也不追加搜索。

**影响：** 当第一次工具调用拿到的上下文不足以确定说话人时（如对话发生在角色刚出场、名字还没出现），模型缺乏"再往前搜"的主动性，转而依赖不可靠的辅助信息或猜测。

## 四、修复方案

### P0 — 立即修复（预期消除 ~60 个错误）

#### 修复 1：重写"非人物发声"规则

**文件：** `src/run_label.py` LabelerAgent.label() 的 system_prompt

**改动：** 将规则 1 和规则 3 的措辞从"互斥"改为"互补"：

```
当前（问题版本）：
 规则1: "如果原文中一个角色只有泛称，不要自己起名字"
 规则3: "无法判断时标注为'非人物发声'"

改为：
 规则1: "使用原文中实际出现的称呼。泛称（村民、骑士、采购员、追兵等）
         本身就是有效的答案——不要因为角色没有具体名字就标'非人物发声'。
         仅当对话明确是内心独白、拟声词、或确实找不到任何说话人线索时，
         才标注为'非人物发声'。"
```

关键：在规则 1 中**直接列举泛称角色作为有效答案示例**，消除模型"泛称=非人物发声"的错误推理路径。

#### 修复 2：修复 `update_character` 调用顺序 + 过滤"非人物发声"

**文件：** `src/run_label.py:887-890`

```python
# 当前（有问题）：
self.char_state.update_character(speaker, line_num, dialogue, context="")  # 无条件注册
# ... 后面才解析 state_update ...

# 修复后：
if speaker != "非人物发声":  # 过滤
    self.char_state.update_character(speaker, line_num, dialogue, context="")
```

**额外防御：** 在 `CharacterState.add_character()` 中增加黑名单检查。

### P1 — 高优先级（预期消除 ~15 个错误）

#### 修复 3：修复 `NEW character` 解析

**文件：** `src/run_label.py` 的 `parse_state_update()` 方法

**改动：** 在创建新角色时，截断 `——` 及之后的所有描述文本：

```python
# 正则从：
match = re.search(r"NEW character:\s*(.+)", text)

# 改为：
match = re.search(r"NEW character:\s*([^—\n]+)", text)
name = match.group(1).strip()
# 额外验证：name 不能是"无"、空字符串或纯描述文本
```

#### 修复 4：减少过度免责声明

**当前：** System Prompt + User Message 中共 **5 处**"仅供参考、可能有误、不要盲信"的免责声明，造成模型对辅助信息过度不信任。

**改动：** 合并为 1 处结构化的风险说明，区分两类信息：

```
辅助信息可信度分级：
  ✅ 可靠（直接可用）：
    - 行号和原文导航中的内容（来自 novel.txt，100% 准确）
    - 短期记忆中上一轮的原文对话文本（100% 准确）
  ⚠️ 参考（需要验证）：
    - 短期记忆中的说话人标注（可能是之前轮次的错误）
    - 长期记忆中的角色描述（压缩过程中可能丢失细节）
    - 角色状态表中的别名映射（可能不完整）
```

---

### P2 — 中优先级（预期消除 ~20-30 个错误）

#### 修复 5：修正说话人判断优先级（不强化交替规律）

**用户重要反馈：** 交替规律在很多场景不适用——同一个角色可能连续说话。如果 Prompt 过于强调交替规律，模型会错误地认为"上一轮说话的人现在不可能说"。

**改动策略：** Prompt 中的规则 4 从"强调交替规律"改为**"叙事标记优先"**：

```
多人对话中的归属判断（优先级从高到低）：
  1. 叙事标记是唯一权威信号：
     "XX说"、"XX开口道"、"XX喊道"、"XX回答" → 直接确定说话人
  2. 同一人可能连续说话——叙事中若没有"他又说"等标记，
     可能是同一角色的连续台词，不要强行假设交替
  3. 仅在叙事完全没有标记说话人时，交替规律才作为推测参考
  4. 如果内容逻辑明确（问→答、命令→服从），可作为辅助信号
```

#### 修复 6：鼓励深度搜索

**当前：** Prompt 中有"阶段2-深度搜索"的指导，但模型几乎不执行（仅 1 次 2 轮工具调用）。

**改动：** 在 User Message 末尾加入显式提醒：

```
如果第一次读取的上下文不足以确定说话人（例如角色刚出场、名字未出现），
请务必再次调用 read_novel_lines 往前搜索更远的范围来确认。
不要依赖短期记忆中的标注结果来做最终判断。
```

## 五、预期效果

| 修复组合 | 预期消除错误数 | 预期准确率 |
|---|---|---|
| 当前 (v2.1) | — | 50.1% |
| + P0（规则重写 + Bug修复） | ~60-80 | ~55-56% |
| + P1（解析 + 免责优化 + "罗伦斯默认"偏差对抗） | ~40-60 | ~58-61% |
| + P2（优先级 + 深度搜索） | ~50-80 | ~62-67% |
| **合计（乐观）** | **~150-220** | **~62-67%** |

**核心约束：** 罗伦斯↔赫萝交换（300 次，44.7%）是最大的单一错误源，且与 Qwen3:32B 的推理能力相关。即使 Prompt 优化到极致，这个数字也不太可能降到 100 以下。按乐观估计（减少 150 个），准确率上限约 **67%**。

如需突破 70%+，需要更根本的改变：
- 升级模型（如 Qwen3:70B / DeepSeek-V3）
- 在 Prompt 中加入针对性的"反默认罗伦斯"偏见纠正规则
- 增加每轮 ShortMem 轮数（当前 15 → 25-30）以提供更完整的对话历史
- 考虑在 Boss 端做后处理校验（如连续 N 轮都是同一说话人 → 触发怀疑）
