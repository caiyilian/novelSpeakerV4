# 记忆机制重新设计方案

> 日期：2026-06-22
> 动机：当前 LongMem 是自由文本摘要，质量差、有幻觉、不可验证。需要向 AI 写小说项目的结构化记忆方案靠拢。

---

## 一、当前记忆的问题

```
当前流程（每轮）：
  Labeler 标注一句 → ShortMem 存 15 轮原文
                   → CharacterState 自动注册 speaker（含"非人物发声"）
                   → 每 5 轮，LongMem 让模型写一段摘要

问题：
  LongMem 产出自由文本 → 无法验证、无法查询、无法纠错
  CharacterState 纯正则解析 → 无验证、"非人物发声"被注册、"寄宿在麦子里的狼"被当独立角色
  记忆和标注耦合 → Labeler 既要标注又要维护状态，职责过重
```

---

## 二、新架构：三层记忆 + 结构化事实库

```
                    ┌─────────────────────────┐
                    │      Boss（调度）        │
                    └────────────┬────────────┘
                                 │
        ┌────────────────────────┼────────────────────────┐
        │                        │                        │
   ┌────┴─────┐           ┌──────┴──────┐          ┌─────┴──────┐
   │ Labeler  │           │ FactCurator │          │ MemAuditor │
   │（标注）   │           │（事实库维护）│          │（幻觉检查） │
   └────┬─────┘           └──────┬──────┘          └─────┬──────┘
        │                        │                        │
        │               ┌────────┴────────┐               │
        │               │   FactBase      │               │
        │               │ （结构化事实库）  │◄──────────────┘
        │               └────────┬────────┘
        │                        │
   ┌────┴─────┐           ┌──────┴──────┐
   │ ShortMem │           │  SceneMem   │
   │（原文缓存）│           │（场景上下文） │
   └──────────┘           └─────────────┘
```

### 三层记忆

| 层 | 名称 | 存储内容 | 更新频率 | 大小 |
|---|------|---------|:---:|---|
| L1 | **ShortMem** | 最近 25-30 轮对话原文 + 说话人 | 每轮 | ~4,000 token |
| L2 | **SceneMem** | 当前场景描述、在场角色、场景事件 | 场景切换时 | ~500 token |
| L3 | **FactBase** | 结构化角色事实、关系、情节节点 | 每 10 轮（由 FactCurator） | ~2,000 token |

### 与旧设计的对比

| 维度 | 旧 LongMem + CharacterState | 新 FactBase |
|---|---|---|
| 格式 | 自由文本摘要 + 简单 JSON map | 完全结构化 JSON |
| 可验证性 | ❌ 文本无法自动检查 | ✅ 每个事实关联原文行号 |
| 别名处理 | 简单 map，无证据追踪 | alias 列表 + identity_revealed 行号 + 证据链 |
| 关系网络 | 简单 key-value | typed relations + confidence + since_line |
| 场景追踪 | ❌ 无 | ✅ 场景列表 + 角色归属 + 关键事件 |
| 情节主线 | ❌ 无 | ✅ plot_threads + key_milestones |
| 维护方式 | Labeler 顺手写 `<state_update>` | 独立 FactCurator Agent 专门维护 |
| 幻觉检测 | ❌ 无 | ✅ MemAuditor 检查 + 事实证据交叉验证 |

## 四、新 Agent 设计

### FactCurator（事实库维护官）

```
职责：每 10 轮运行一次，维护 FactBase

输入：
  - 当前 FactBase（JSON）
  - 最近 10 轮的 ShortMem 原文
  - Labeler 的标注结果

输出：
  - FactBase 增量更新（JSON patch）
  - 新角色注册、别名确认、关系更新、场景更新
  - 冲突标记（如发现与已有事实矛盾的新信息）

System Prompt 核心约束：
  "你是角色数据库维护助手。你的任务是新增/更新角色事实，不是编写小说摘要。
   每个更新必须引用原文行号作为证据。
   发现矛盾时标记 conflict 而不是覆盖。
   不要编造任何原文中找不到的信息。"
```

### MemAuditor（幻觉检查官）

```
职责：在 FactCurator 更新后运行，检查新增事实的质量

输入：
  - FactCurator 的增量更新
  - 对应的原文证据行

输出：
  - pass / flag / reject 判定
  - 具体问题描述（如"声称赫萝是狐娘，但原文 L275 明确说'咱可是崇高无上的狼'"）

检查项：
  1. 种族/物种是否与原文一致
  2. 身份描述是否有原文支持
  3. 别名映射是否合理
  4. 关系声明是否有证据
```

## 五、Labeler 的职责简化

当前 Labeler 既要标注又要维护状态（写 `<state_update>`），负担过重。

### 新设计：Labeler 只负责标注

```
Labeler 输出：
  <answer>说话人</answer>
  <reason>判断依据（引用行号）</reason>
  <discovery>（可选）发现新角色/别名/关系时标记</discovery>

<discovery> 示例：
  NEW_NAME: 第222行，女孩自报姓名"赫萝"
  NEW_ALIAS: 第275行，赫萝自称"崇高无上的狼"
  SCENE_CHANGE: 第1127行，从市场切换到酒吧
```

- `<discovery>` 只做**标记**，不做**执行**
- FactCurator 读取 `<discovery>` 标记 + 原文证据 → 决定是否写入 FactBase
- Labeler 不需要理解 "UPDAtE character / NEW character" 的语法
- 避免了当前 "NEW character: 寄宿在麦子里的狼（原文揭露）" 这种语法错误

## 六、预期效果

| 维度 | 当前 v2.1 | 新设计 |
|---|---|---|
| LongMem 幻觉 | "赫萝是狐娘""赫萝是精灵" | FactCurator 有证据约束 → 输出结构化事实 |
| CharacterState 污染 | "非人物发声"被注册、"寄宿在麦子里的狼"独立条目 | MemAuditor 过滤 + FactCurator 不创建无效角色 |
| 别名追踪 | 简单 map，无证据 | 每个别名关联原文行号，可追溯 |
| 场景感知 | 无 | 场景列表 + 角色归属 |
| 角色关系 | 无（关系字段为空） | typed relations with confidence |
| Labeler 负担 | 既要标注又要写 `UPDATE` 语法 | 只标注 + 可选 `<discovery>` 标记 |
| 可审计性 | 无 | 每个事实可回溯到原文证据 |
| 记忆 Token 占用 | ~1,000 token（LongMem） | ~2,000 token（FactBase，但信息量 ×10） |

## 七、实现路线

### Phase 1：停用有害组件（立即）
- 停用 LongMem 压缩（不再产生幻觉摘要）
- 过滤"非人物发声"不入 CharacterState
- 修复 NEW character 解析

### Phase 2：FactBase 基础设施
- 定义 FactBase JSON Schema
- 实现 FactBase 的读写 API
- 扩大 ShortMem 到 25-30 轮（上下文充裕）

### Phase 3：FactCurator Agent
- 实现 FactCurator 的 System Prompt 和调用逻辑
- 接入每 10 轮触发
- 测试结构化输出的准确性

### Phase 4：MemAuditor Agent
- 实现幻觉检查逻辑
- 接入 FactCurator 之后运行
- 加入自动修复（检测到种族错误 → 回退修正）

### Phase 5：简化 Labeler
- 去掉 `<state_update>` 标签
- 替换为可选 `<discovery>` 标记
- Boss 端解析 `<discovery>` 而非执行状态更新
