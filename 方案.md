# 小说对话角色标注 — 多 Agent 记忆架构方案

> 基于 Qwen3:32B 模型，服务器 `172.31.102.237:11434`
> 目标：对《狼与香辛料》第一卷进行逐条对话说话人标注，追求质量优先

---

## 一、问题分析

### 标注任务

将小说中形如 `「对话内容」` 的对话，标注上对应的说话角色名：

```
输入  (novel.txt)
「这是最后一件了吧？」
「嗯，这里确实有……七十件。多谢惠顾。」

输出  (labeled.txt)
村民
罗伦斯
```

### 标注格式

| 格式 | 含义 | 示例 |
|------|------|------|
| `【A】` | 唯一答案 | `【罗伦斯】` |
| `【A\|B\|C】` | 多个可接受答案 | `【赫萝\|贤狼赫萝】` |
| `【A\|非人物发声】` | 非对话也可接受 | `【罗伦斯\|非人物发声】` |

### 标注难点

| 难点 | 示例 | 说明 |
|------|------|------|
| 隐含说话人 | 第166行「……嗯唔？」 | 无标注，需从上下文推断 |
| 长距离指代 | 第96行「村民\|农夫」 | 需回溯前文确认同一人 |
| 身份揭示 | 第222行「咱的名字是赫萝」 | 从"女孩"到"赫萝" |
| 多人同名 | 第111行村民喊"赫萝" | 既是角色名也是被喊的名字 |
| 内心独白 | 第158行「……」 | 答案标为 `非人物发声` |

### 核心挑战

逐条标注时，**相邻对话之间没有记忆**——标注第100条时，不知道前面90条发生了什么。这会导致：
- 角色身份变化无法追踪（"女孩" → "赫萝"）
- 角色关系丢失（罗伦斯和赫萝的关系进展）
- 重复推理（每次都重新推断角色身份）

## 二、架构设计

### 整体架构

```
                          ┌──────────────┐
                          │  novel.txt   │
                          │  (原始小说)   │
                          └──────┬───────┘
                                 │
                    ┌────────────┼────────────┐
                    │            │            │
              ┌─────┴─────┐ ┌───┴────┐ ┌─────┴──────┐
              │ ShortMem  │ │  Boss  │ │  LongMem   │
              │ 短期记忆   │ │ 主Agent │ │  长期记忆   │
              └─────┬─────┘ └───┬────┘ └─────┬──────┘
                    │           │            │
                    └─────┬─────┘────────────┘
                          │
                    ┌─────┴─────┐
                    │  Labeler  │
                    │  标注Agent │
                    │  + 工具调用 │
                    └─────┬─────┘
                          │
                    ┌─────┴─────┐
                    │ labeled.txt│
                    │  (标注结果) │
                    └───────────┘
```

### 各 Agent 职责

#### Boss（主 Agent）

- **角色**：任务调度 + 质量把控
- **职责**：
  1. 读取 `labeled.txt` 获取已标注数量
  2. 调用 `get_dialogue.py` 获取下一条待标注对话
  3. 向 ShortMem 和 LongMem 获取记忆摘要
  4. 将任务分发给 Labeler
  5. 接收 Labeler 结果，写入 `labeled.txt`
  6. 定期触发 LongMem 压缩
- **上下文消耗**：极低（只接收汇总，不接触原始文本）

#### Labeler（标注 Agent）

- **角色**：核心标注执行者
- **职责**：
  1. 接收当前对话 + 上下文片段
  2. 接收短期记忆和长期记忆摘要
  3. 分析说话人身份
  4. 如有需要，调用工具回读小说原文确认线索
  5. 输出标注结果 + 本次标注摘要
- **工具能力**：
  - `read_novel_lines(start, count)` — 读取小说任意行范围
  - 不限调用次数，按需使用
- **上下文消耗**：中等（当前对话 + 上下文 + 记忆摘要 + 工具调用结果）

#### ShortMem（短期记忆 Agent）

- **角色**：维护最近 N 轮的标注历史
- **职责**：
  1. 存储最近 N 轮的标注摘要
  2. 每轮更新：加入最新标注结果，淘汰最旧的
  3. 为 Labeler 提供近期上下文
- **存储格式**：
  ```
  [短期记忆 - 最近5轮]
  #23: 村民 → 罗伦斯 | 卖皮草70件
  #24: 罗伦斯 → 村民 | 感谢
  #37: 罗伦斯 | 看到修道院有人挥手
  #41: 罗伦斯 | 发现是骑士
  #43: 骑士 → 罗伦斯 | 盘问身份
  ```
- **上下文消耗**：低（约 1-2K tokens）

#### LongMem（长期记忆 Agent）

- **角色**：渐进压缩短期记忆，提炼关键信息
- **职责**：
  1. 接收短期记忆历史
  2. 压缩提炼：保留关键信息，丢弃细节
  3. 为 Labeler 提供全局上下文
- **保留内容**：
  - 角色关系图谱（谁是谁、什么关系）
  - 当前主线情节（正在去哪里、目的是什么）
  - 关键设定（赫萝的身份、罗伦斯的身份）
  - 角色别名映射（赫萝 = 贤狼赫萝 = 女孩 = 丰收之神）
- **丢弃内容**：
  - 具体对话内容
  - 交易细节
  - 路人名字
  - 已过时的临时信息
- **压缩策略**：类似"指数移动平均"——越久远的信息压缩率越高
- **压缩频率**：待测试验证（建议初始值：每 5 轮压缩一次）
- **上下文消耗**：低（约 2-3K tokens）

## 三、工作流程

### 单轮标注流程

```
Step 1: Boss 读取 labeled.txt → 获取已标注数量 N
Step 2: Boss 调用 get_dialogue.py → 获取第 N+1 条对话
Step 3: Boss 向 ShortMem 请求 → 获取最近 N 轮标注摘要
Step 4: Boss 向 LongMem 请求 → 获取角色关系和主线情节
Step 5: Boss 组装任务 → 分发给 Labeler
Step 6: Labeler 分析 → 必要时调用工具 read_novel_lines
Step 7: Labeler 输出 → 标注结果 + 本次标注摘要
Step 8: Boss 接收结果 → 写入 labeled.txt
Step 9: ShortMem 更新 → 加入最新标注摘要
Step 10: 每隔 N 轮 → LongMem 压缩短期记忆
Step 11: 回到 Step 1
```

### 输入输出示例

**Labeler 的输入：**
```

[上下文片段 - 第160-170行]

166: 「……嗯唔？」
167: 随着罗伦斯的声音，女孩闭着双眼...

[短期记忆 - 最近5轮]
```

## 四、上下文预算分配

### Labeler 上下文（主要消耗者）

| 内容 | 预估 Token |
|------|-----------|
| 当前对话原文 | ~200 |
| 上下文片段（±20行） | ~800 |
| 短期记忆摘要 | ~1,000 |
| 长期记忆摘要 | ~1,500 |
| 工具调用结果（按需） | ~0-2,000 |
| Labeler 输出 | ~300 |
| **小计** | **~3,800-5,800** |

### Boss 上下文（极低消耗）

| 内容 | 预估 Token |
|------|-----------|
| 调度指令 | ~200 |
| 各 Agent 汇总 | ~500 |
| **小计** | **~700** |

### ShortMem 上下文

| 内容 | 预估 Token |
|------|-----------|
| 最近 N 轮标注摘要 | ~1,500 |
| 更新指令 | ~200 |
| **小计** | **~1,700** |

### LongMem 上下文

| 内容 | 预估 Token |
|------|-----------|
| 短期记忆历史 | ~2,000 |
| 压缩指令 | ~300 |
| **小计** | **~2,300** |

| 项目 | Token |
|------|-------|
| Labeler | ~5,000 |

## 五、记忆更新机制

### 短期记忆更新（每轮）

```python
# 伪代码
short_memory.append({
    "line": 166,
    "speaker": "赫萝",
    "summary": "被罗伦斯叫醒，回应「……嗯唔？」"
})

# 超过 N 轮时，淘汰最旧的
if len(short_memory) > N:
    short_memory.pop(0)
```

## 六、工具调用策略

### Labeler 可用工具

```python
{
    "name": "read_novel_lines",
    "description": "读取 novel.txt 的指定行范围",
    "parameters": {
        "start": "起始行号(1-based)",
        "count": "读取行数"
    }
}
```

### 工具调用原则

1. **上下文片段优先**：Boss 已经提供了 ±20 行上下文，Labeler 应先基于此分析
2. **按需调用**：仅在以下情况调用工具：
   - 需要确认角色身份（如"女孩"是谁）
   - 需要回溯前文线索（如角色首次登场时的描述）
   - 需要确认对话连续性（如说话人是否切换）
3. **不限次数**：只要上下文窗口允许，Labeler 可以多次调用工具
4. **跳跃式搜索**：不逐行扫描，优先大范围定位关键信息

### 工具调用示例

```
Labeler 分析第166行：
1. 上下文显示说话人是"女孩"，但不知道名字
2. 调用 read_novel_lines(200, 30) → 搜索名字出现的位置
3. 在第222行找到"咱的名字是赫萝"
4. 输出标注结果：赫萝
```

## 七、验证机制

### 验证原则

- **Ollama 大模型不能看到 answers.txt**（防止作弊）
- 验证由独立脚本在标注完成后统一执行
- 验证结果仅用于评估，不反馈给标注 Agent

### 验证脚本逻辑

```python
# 伪代码
def validate(labeled.txt, answers.txt):
    labeled = read_labeled()
    answers = read_answers()
    
    total = 0
    correct = 0
    wrong = []
    
    for i, (label, answer) in enumerate(zip(labeled, answers)):
        total += 1
        if label in answer.split("|"):
            correct += 1
        else:
            wrong.append({
                "line": i + 1,
                "expected": answer,
                "got": label
            })
    
    accuracy = correct / total
    return accuracy, wrong
```
